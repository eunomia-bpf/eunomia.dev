# AI Agent 指令文件里，64% 的语句是策略

一个 AI coding agent 执行 `git commit` 时，内核看到的只是普通进程在写普通文件。仓库里的 CLAUDE.md 却明确要求“提交前必须跑完整测试套件”，而这个 agent 在上次测试之后又修改了源码。这句话来自真实的 AI agent 指令文件，但当前技术栈没有任何一层能够保证它得到执行。

[ActPlane 论文](https://arxiv.org/abs/2606.25189)没有从假设的攻击面出发，而是统计开发者已经写进 CLAUDE.md 和 AGENTS.md 的要求，再分析这些要求如何才能变成可执行规则。研究共分析 2116 条语句，其中 64% 属于策略，83% 的策略涉及系统可观测行为，而 74% 的系统可观测策略依赖无法静态预定义的上下文。

## 2116 条语句如何呈现 AI Agent 指令的真实面貌

讨论 AI agent 安全时，人们通常先列威胁模型。这项研究换了一个更贴近仓库维护者的起点。开发者已经在指令文件里写了大量“必须做”和“不得做”，那么这些要求究竟是什么，又需要怎样的检查才能真正生效？

数据集收录了 64 个包含 CLAUDE.md 或 AGENTS.md 的热门仓库。以 2026-05-23 的快照计算，这些仓库的 GitHub star 中位数为 20K，共提供 84 份指令文件和 2116 条独立语句。此前的指令文件研究往往以整份文件或章节标题为单位，ActPlane 则逐条分析每一个语句。

这种粒度让研究可以分别回答三个问题。指令文件里的内容主要是行为策略，还是用于帮助 agent 理解项目的描述？哪些策略需要操作系统级强制执行，又需要哪一类检查？当自然语言策略要变成具体规则时，还缺少哪些项目或任务上下文？

语句提取采用两遍 LLM agent 辅助流水线。流水线记录每条语句在源文件中的行范围，并标注内容类型、主题、强制执行层级和上下文需求。验证脚本随后检查源文件是否得到完整覆盖，提取范围是否逐字匹配。Claude 和 Codex 两个独立 agent 对结果进行交叉检查，人工标注者又独立审核了 100 条分层抽样语句，确认这些标签正确。

统计结果首先改变了我们对指令文件用途的认识。2116 条语句中，64% 会要求、禁止或根据条件约束某个具体的 agent 行为，因此属于策略（policy）；其余 36% 是架构说明、项目背景等描述性上下文。

不同仓库的写法差异很大，策略密度从 0% 到 97% 不等，70.1% 的仓库拥有比描述更多的策略。若按行数统计，这个结果会被掩盖。描述平均占 6.8 行，策略平均只有 3.6 行，因此按行计算时策略比例仅为 49%。语句级分析能够避免被篇幅差异误导。

研究还将每条语句归入 12 个主题类别，这套分类改编自此前的指令文件研究，但应用到了语句而非文件层面。开发流程和实现细节中的策略比例最高，分别达到 87% 和 85%。架构类内容通常用于记录目录布局与设计结构，其中只有 23% 属于策略。

![64 个含 CLAUDE.md 或 AGENTS.md 的仓库中每个仓库的策略比例](imgs/actplane-empirical_rq1_policy_fraction.png)

数据集里的五条真实语句看起来都像简短的仓库要求，背后需要的检查方式却完全不同：

| 语句 | 强制执行层级 | 上下文 |
|---|---|---|
| S4: "Never push to main directly." | 单事件（per-event） | 自包含（self-contained） |
| S5: "Never modify upstream source code." | 单事件 | 项目级 |
| S6: "Run the full test suite before committing." | 跨事件（cross-event） | 项目级 |
| S7: "Data read from .env must not reach the network." | 跨事件 | 项目级 |
| S8: "Do not update dependencies without approval." | 单事件 | 任务级 |

S4 可以在一次操作发生时判断。S6 必须知道源码何时修改、测试何时运行，S7 还要追踪数据来源，S8 则取决于当前任务是否得到批准。它们虽然都以祈使句出现在同一种文件里，却对应着不同的强制执行问题。

## 大多数策略可以观测，真正困难的是补齐上下文

研究按照“强制执行瀑布”把每条策略放入第一个匹配的层级。纯语义（semantic-only）策略约束推理、沟通或输出风格；内容检测（content inspection）需要检查文件内容上的条件；单事件策略只需判断一次命令执行、文件访问或网络连接；跨事件策略则依赖多个操作之间的时间顺序或数据来源。内容检测、单事件和跨事件合称系统可观测（system-observable）。

1361 条策略中，只有 17% 属于纯语义，其余 83% 原则上都能通过系统级监测器判断。其中 38% 需要内容检测，29% 匹配单个操作系统事件，16% 需要维护跨事件状态。全部跨事件策略中，39.5% 来自开发流程，这与仓库经常约束测试、代码生成、评审和发布顺序相吻合。

![强制执行瀑布：纯语义、内容检测、单事件和跨事件在 1361 条策略上的分布](imgs/actplane-empirical_waterfall_enforcement.png)

跨事件策略反复出现四种形态。时序排序（temporal ordering）要求一个动作发生在另一个动作之后，例如提交前必须重新运行测试。跨文件一致性（cross-file consistency）把不同工件的修改联系起来，例如行为变化后同步更新文档。多步工作流（multi-step workflow）会在发布清单中加入验证门，前一步完成后才能继续。条件触发（conditional trigger）只在相关前提出现后产生义务，例如规范变化后必须更新 SDK。

单个事件无法回答这些问题。强制执行层必须记住哪些命令运行过、是否成功、先后顺序如何，以及此后又发生了哪些变化。此类需求并不罕见，81% 的仓库至少包含一条跨事件策略，43% 的仓库同时覆盖全部四个强制执行层级。

即便一个行为能够被系统观测，规则所需的具体对象通常也没有写全。1127 条系统可观测策略中，只有 26.4% 是自包含的，64.2% 需要项目上下文，另有 9.4% 依赖任务上下文。

S5 能直观说明这个问题。“不得修改上游源码”只需在一次文件写入时作出决定，但系统仍然要先知道仓库中哪些路径属于“上游源码”。“除非用户明确要求”或“未经批准不得操作”一类条件则必须结合当前任务，才能确定这一次行为是否允许。

跨事件策略把两类困难叠在了一起。它们有 95% 依赖上下文，其中 77% 需要项目信息，19% 需要任务信息；相比之下，内容检测策略的上下文依赖比例为 58%。“提交前跑测试”对开发者已经足够清楚，规则引擎却还要解析具体测试命令、需要监视的源码目录，并判断测试到底成功了，还是仅仅运行过。

![上下文瀑布：自包含、项目上下文和任务上下文在 1127 条系统可观测策略上的分布](imgs/actplane-empirical_waterfall_context.png)

固定的静态规则只能直接覆盖自包含部分。剩下的策略必须先结合仓库与当前任务完成解释，再落到明确的命令、路径、事件和条件上。

## 这些指令文件对强制执行层提出了什么要求

把规则放进 prompt 仍然有价值，因为模型能够在上下文中读到项目知识。但这种方式最终依赖模型主动遵守，既会受到 prompt 注入影响，也会在长任务中与用户指令争夺注意力。独立 agent 或 LLM 守卫可以检查 prompt、回复和行为轨迹，但判断仍然带有概率性。

工具调用守卫与应用级信息流控制系统能够在 harness 边界作出确定性判断，不过它们只能看到经过 harness 中介的请求。工具一旦启动子进程、调用 shell 或运行编译后的程序，系统实际产生的效果可能与最初的工具请求完全不同。

例如，agent 可以先写一个包含 `subprocess.run(["git", "push"])` 的 Python 程序，再执行这个程序。工具层批准的是“运行 python script.py”，真正产生影响的 `git push` 出现在新进程内部，从未作为工具调用暴露出来。

seccomp、AppArmor、Landlock 和 Tetragon 等操作系统机制能够看到更底层的资源访问，但它们通常要求预先写好静态策略。资源权限也不等同于开发者在指令文件中描述的行为。操作被拒绝时，agent 往往只能收到不透明的 `EPERM`，既不知道违反了哪条仓库规则，也不知道应该怎样恢复。

实证结果由此划出了一条职责边界。项目和任务上下文靠近 agent，因此自然语言策略需要由 agent 参与实例化；事件顺序和数据流位于工具边界之下，因此实例化后的规则需要在操作系统层面得到确定性执行。

这条边界带来两项设计要求。策略规范既要便于 agent 编写，也要具体到足以由操作系统执行，违规反馈还必须说明原因，让 agent 能够调整计划。强制执行本身也要保持安全、隔离和高效，agent 编写的策略不能削弱更高权限设置的约束，不能影响其他 agent 的策略域，也不能显著拖慢正常工作负载。

## 如何把实证分类变成具体规则

这项研究没有停在分类统计上，而是通过 [ActPlane](https://github.com/eunomia-bpf/ActPlane) 检查这些策略类型能否转成规则，并在真实执行路径上得到强制执行。

![ActPlane 总览：离任务最近的 agent 编写具体策略 DSL，由内核编译并执行](imgs/actplane-illustration.png)

一条 ActPlane 规则由五部分组成。来源声明确定治理对象，目标指定 exec、write 或 connect 等操作，效果说明如何响应，可选的时序门加入历史条件，原因字符串则负责返回语义反馈。论文用下面这条规则贯穿说明：

```
kill exec "git" "commit" unless after exec "go" "test" exits 0
```

除非 `go test` 在最近一次相关源码修改之后成功退出，否则这条规则会终止 `git commit`。示例中省略的原因字段会在规则触发时给 agent 一段结构化解释。

三种执行效果形成了不同强度。阻断（block）会在操作前同步拒绝系统调用，不留下 TOCTOU 窗口，agent 随后可以换一条路径。终止（kill）在操作开始后杀掉进程，避免 agent 在一次阻断后改用其他通道。通知（notify）只给出引导，不会停止操作。约束使用阻断或终止，指令则使用通知。

时序门表达的是事件顺序，而非某个时刻的静态条件。`after ... since ...` 要求一个事件发生在另一个事件之后，因此最近一次源码修改之前的测试不能满足提交条件。`exits N` 用来区分成功和失败的执行结果。谱系门（lineage gate）检查进程祖先关系，可以把某项操作限制在指定进程树中。

信息流标签（information-flow label）对应实证分类中的另一类要求。标签沿 fork、exec、read、write 和 connect 传播，并且保持单调。进程一旦读取带标签的对象，就不能移除获得的标签。读取 `.env` 的进程会继承该文件的来源标签，之后连接外部端点时，匹配这个标签的规则便会触发并阻断连接。数据集中的 S7“从 `.env` 读取的数据不得到达网络”由此成为可执行的跨事件规则。

策略权限由时序信任边界保护。agent 启动前加载的规则拥有更高权限，agent 无法修改。Agent 及其子 agent 可以在子域里添加新规则或收窄已有规则，却不能削弱、删除或禁用继承下来的约束。

运行时策略增量通过 ring buffer 送入内核权限检查器。每项变更都要按照域层级通过校验后才能激活。因此，信任计算基础只包含内核强制执行引擎和高权限策略，边界以下的执行均视为不可信。权限检查器完全运行在内核空间，即便用户空间 agent 被攻陷，也不能越过所在域的权限修改活跃规则集。

标签的单调性会在长会话中带来过标记风险。一个进程读取几十个配置文件和源文件后，可能积累大量标签，导致后续写入或连接不断匹配规则。ActPlane 会在生成新的子进程时清除继承标签，把污染累积限制在单个进程的生命周期内，而不是延续到整个会话。

607 条翻译策略还展示了真实指令会用到哪些语言特性。全部子句中，66% 使用通知，29% 使用阻断，只有 5% 使用终止，因此观测比禁止更常见。挂载点主要集中在代码执行与文件修改，60% 使用 exec，37% 使用 write，网络和清理操作各不到 1%。跨事件结构同样占有相当比例，28% 的策略使用 `after/since` 时序门，另有 214 条使用 `unless` 表达例外。

用户空间编译器和运行器约有 3.2K 行 Rust 代码，eBPF 强制执行引擎约有 1.8K 行 BPF C 代码。BPF-LSM hook 负责操作前的阻断判断，tracepoint 负责观测和操作后的终止效果。逐对象 BPF map 以 64 位掩码保存标签，传播只需一次按位 OR。引擎最多支持 128 条并发规则，而研究中规模最大的仓库包含 66 条策略。部署架构和机制细节可参阅 [ActPlane：把 Agent Harness 强制执行下沉到内核 eBPF](https://eunomia.dev/zh/blog/2026/05/31/actplane-pushing-agent-harness-enforcement-down-to-kernel-ebpf/)。

## 当规则遇到真实执行路径

第一个实际问题是，agent 能否按照数据集呈现的规模翻译仓库指令。一个 Codex agent 在首次或第二次尝试中，将全部 607 条可由操作系统执行的策略编译成 ActPlane 规则，其中只有 2 条因为语法错误需要重试。

整个过程使用 7 个子 agent，其中 4 个并行运行，34 分钟内把 607 条策略展开为 1283 行规则。翻译消耗 1.7M 输入 token 和 177K 输出 token，平均每条策略约 $0.028。论文以典型软件工程师成本估算，人工编写每条规则约为 $11。多数策略的结构并不复杂，74% 最多包含两个强制执行子句，第 95 百分位仍低于 152 token。

第二个问题是，结合上下文的操作系统级强制执行能否改变 agent 的实际结果。在包含 190 条轨迹和 38 条实证研究规则的决策合规基准上，ActPlane 的决策合规率（Decision Compliance Rate, DCR）达到 75.8%：

| 系统 | DCR |
|---|---|
| ActPlane | 75.8% |
| ActPlane-opaque（无反馈） | 53.7% |
| FIDES（工具级信息流控制） | 48.9% |
| Prompt-filter | 48.4% |
| Tool-regex | 45.3% |

每条规则对应五条轨迹。两条合规轨迹用于检查误报，其中一条执行接近策略边界但仍被允许的行为，另一条执行技术上合规却很像违规的操作。另外三条是违规轨迹，复现部署中的 coding agent 所采用的执行路径。

直接违规通过工具调用触发被禁止的操作。脚本违规把编写和执行拆开，将操作放进子进程。隐藏违规则把副作用塞进辅助工件，并用一个中性的入口点启动。

各系统在合规轨迹和直接违规上的表现接近，差距出现在执行跨过工具边界之后。114 条违规轨迹中，ActPlane 正确解决了 86 条，基线只能解决 27 到 44 条，提升达到 2.0 到 3.2 倍。脚本轨迹和隐藏轨迹构成主要优势，因为违规效果发生在子进程内部或中性入口点背后，工具调用拦截看不到这些行为。

语义反馈决定了强制执行会带来合规，还是触发盲目重试。完整 ActPlane 正确解决 86 条违规轨迹，无反馈版本只解决 27 条，相差三倍。加入语义反馈后，恢复率达到 97.7%，没有反馈时仅为 31.4%。agent 得知缺少什么状态后，可以调整计划并完成任务；若只收到不透明的拒绝，往往会换一条路径重复同一个被禁止的操作。

性能评估采用策略已经加载、但没有任何规则触发的空命中配置。第一个工作负载重放包含 68 个工具动作和 20 个 Bash 子进程的 agent 轨迹套件，第二个使用 `defconfig` 加 `vmlinux` 和 `make -j24` 构建 Linux 内核。加载 32 条活跃规则时，ActPlane 给 agent 轨迹增加 1.9% 开销，给内核构建增加 6.5%；规则增至 100 条后，开销仍低于 8.4%。

微基准进一步定位了单次系统调用的成本。随着规则数量变化，fork 和 exec 增加的绝对延迟最高，范围为 3 到 69 微秒，而两者的原生延迟分别为 49 和 248 微秒。加载 100 条规则时，open 达到 13.4 微秒，write 和 connect 仍低于 1 微秒。

若把一次工具调用产生的全部系统调用计算在内，ActPlane 的累计开销仍比一次 2 到 10 秒的 LLM 推理小五到六个数量级。策略更新也很快。通过用户空间 ring buffer 提交一条热重载规则后，平均 26.3 微秒即可到达内核排空路径。一次立即发生的 exec 违规在 p50 176.4 微秒内被检测出来，其中已经包含进程启动和事件传递时间。

换用第二个模型后，系统排名保持不变。DeepSeek-Pro V4 的 DCR 最高，达到 77.4%，两个模型设置之间的逐单元一致性得到 Cohen's kappa 0.822。

规则翻译的质量仍会同时影响检测和恢复。规则过窄会漏掉违规，过宽又会匹配合规操作。为检验错误能否修复，论文把每条假阴性轨迹的证据和纠正反馈交给翻译 agent，并允许它修改一次规则。28 条假阴性轨迹中，修订后的规则恢复了 26 条，即 93%，说明这套 DSL 支持迭代完善。

真实编码任务呈现了相同趋势。OctoBench 的 21 个任务子集覆盖七个仓库和 61 条可由操作系统执行的规则。与无强制执行基线相比，ActPlane 将用户查询奖励提高 9.9 分，将实现与测试奖励提高 9.7 分。收益并未局限于合规类型的检查，这表明操作系统级强制执行加语义反馈既能帮助 agent 遵守规则，也能帮助它完成任务。

论文还评估了 361 个 OpenAgentSafety 个人助理任务。ActPlane 在 agent 启动前把 agent 生成的安全策略加载为高权限规则，阻止了 74% 的基线不安全行为，即 106 起不安全结果中的 78 起。

未被阻止的 28 起案例可以分为三类。一类是聊天或语义伤害，不安全行为只是消息，没有可由操作系统观测的工件；一类涉及不安全文件内容，超出了 ActPlane 的主要范围；还有一类属于服务端工件，例如服务容器内的 WebDAV 上传或数据库变更，当前 hook 集无法观测这些效果。

[ActPlane 源码](https://github.com/eunomia-bpf/ActPlane)已在 GitHub 开源。仓库中的 `policies/` 目录收录了 64 个仓库的全部 607 条翻译策略，可以作为其他指令文件编写规则的起点。

## 常见问题

### eBPF 足以解决 AI agent 安全问题吗？

eBPF 能够对文件写入、进程启动和网络连接等操作系统事件进行确定性强制执行，因此可以覆盖 83% 的系统可观测策略。剩余 17% 涉及推理、沟通风格或输出质量，需要由内核之外的 harness 控制处理。

需要读取文件内容的策略占 38%，更适合交给代码检查器和静态分析器，不属于 eBPF 的单事件和跨事件强制执行范围。任务意图、策略权限、内容语义和隔离也仍然需要由内核强制执行点周围的其他层负责。

### 行为基线能替代策略吗？

行为基线通过历史模式判断某个操作是否异常，策略则判断该操作在当前任务中是否得到允许。

一次常规 `git commit` 按照历史基线可以完全正常，同时违反项目规则。陌生的部署端点也可能触发异常告警，但只要用户明确要求了这项操作，它仍然合法。检测与授权解决的是两个不同问题。成熟系统可以先让异常信号提出候选规则，再经过权限感知审查后决定是否把它变成强制执行策略。

### AgentSight 会执行 ActPlane 的策略吗？

不会。[AgentSight](https://eunomia.dev/zh/blog/2025/08/26/agentsight-keeping-your-ai-agents-under-control-with-ebpf-powered-system-observability/) 是系统级分析器和监测器，用来记录 agent 在运行时做了什么。ActPlane 是强制执行引擎，负责阻断或重定向违反已加载策略的操作。

两者承担互补职责。AgentSight 收集的运行时证据可以用于策略审查和完善，但观测与强制执行仍然彼此独立。组织可以先用 AgentSight 发现 agent 经常写入生产配置文件，再将这种模式编码为 ActPlane 规则，在阻止写入的同时解释禁止原因。

### ActPlane 仍然覆盖不了什么？

论文明确划定了几类边界。语义等价操作可能绕过基于字符串的匹配，agent 可以使用自制 Git 客户端，而不是调用 `exec git`，不过底层 `connect` 和 `write` 系统调用仍然可见。协议边界之后的服务端效果也可能逃出当前 hook 集，例如 WebDAV 上传或服务容器内部的数据库变更。

文件内容语义、内核攻陷、CAP_BPF 攻陷和侧通道均不在覆盖范围内。占比 17% 的纯语义策略涉及推理质量、沟通语气或输出格式，需要由 harness 层处理，而不是依靠内核强制执行。

这项研究最独特的贡献仍然是数据集。在 ActPlane 之前，没有研究度量过开发者究竟要求 agent 遵守哪些规则，也没有研究把这些指令映射到不同的强制执行需求。大量规则已经存在于成千上万个仓库的 CLAUDE.md 和 AGENTS.md 中。要让它们真正生效，系统必须把项目与任务上下文转成具体、确定性的检查。[ActPlane 仓库](https://github.com/eunomia-bpf/ActPlane)包含完整实现，[基于 eBPF 的不透明 AI Agent 运行时可观测与强制执行](https://eunomia.dev/zh/blog/2026/05/25/runtime-security-for-ai-agents/)则把内核强制执行放进一个同时涵盖隔离、身份和内容控制的三层安全模型中。
