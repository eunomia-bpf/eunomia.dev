# CLAUDE.md 规则大多是策略，需要 OS 强制执行与上下文

一个 AI coding agent 执行 `git commit`，内核看不到任何异常：熟悉的进程在写熟悉的文件。仓库的 CLAUDE.md 写着「提交前必须跑完整测试套件」，而这个 agent 在上次测试之后又改过源码。这条规则来自真实指令文件数据集，今天栈上没有任何一层在真正执行它。

[ActPlane 论文](https://arxiv.org/abs/2606.25189)度量的是开发者已经写在 CLAUDE.md 和 AGENTS.md 里的策略（policy），逐条分类强制执行这些策略需要什么条件，并验证 OS 级强制执行加上语义反馈（semantic feedback）能否补上这道缝。在整份数据里，64% 的语句是策略，83% 涉及系统行为，74% 依赖无法静态预先定义的上下文。

## 开发者在 CLAUDE.md 和 AGENTS.md 里到底写了什么

讨论 AI agent 安全时，多数人从威胁模型或攻击面入手。这项研究换了一个更朴素的起点：开发者已经告诉 agent 该做什么、不该做什么，要把这些指令变成可执行的规则，到底还缺什么？

语料覆盖 64 个含 CLAUDE.md 与 AGENTS.md 的热门仓库（中位数 20K GitHub star，快照日期 2026-05-23），合计 84 份指令文件、2116 条独立语句。此前的工作往往只在文件或章节标题粒度分析，这里则对每一条语句单独分类，并围绕三个问题展开。指令文件主要是行为策略，还是描述性背景？哪些策略需要 OS 级强制执行，又需要哪类检查？要把策略实例化成具体可执行规则，还需要什么上下文？

语句提取走两遍 LLM agent 辅助流水线，为每条语句记录源码行范围，以及内容类型、主题、强制执行层级、上下文需求四个标签。验证脚本确认了完整源覆盖和逐字 span 匹配，再由独立的 Claude 与 Codex agent 交叉核对。最后对 100 条分层抽样语句做人工独立审核，确认标签正确。

在这 2116 条语句中，64% 是策略：它们要求、禁止或约束某个具体 agent 行为。其余 36% 是描述性上下文，例如架构说明或项目背景。各仓库策略密度差异很大，从 0% 到 97% 不等；70.1% 的仓库里策略条数多于描述。按行统计会掩盖这一分裂，因为描述平均 6.8 行、策略平均 3.6 行，按行数看只有 49% 是策略，所以语句级分析才站得住。

为了看清策略落在哪些关切上，每条语句被分到改编自先前指令文件研究的 12 个主题类别，粒度是语句而不是文件。开发流程与实现细节两类的策略占比最高，分别达到 87% 和 85%。架构类以描述为主，策略只占 23%，因为目录布局和设计摘要构成了这些章节的主体。

![64 个含 CLAUDE.md 或 AGENTS.md 的仓库中每个仓库的策略比例](imgs/actplane-empirical_rq1_policy_fraction.png)

数据集中的五条真实语句，足以看出强制执行需求有多宽：

| 语句 | 强制执行层级 | 上下文 |
|---|---|---|
| S4: "Never push to main directly." | 单事件（per-event） | 自包含（self-contained） |
| S5: "Never modify upstream source code." | 单事件 | 项目级 |
| S6: "Run the full test suite before committing." | 跨事件（cross-event） | 项目级 |
| S7: "Data read from .env must not reach the network." | 跨事件 | 项目级 |
| S8: "Do not update dependencies without approval." | 单事件 | 任务级 |

## 大多数规则系统可观测，最难的是跨事件策略

每条策略按强制执行瀑布（enforcement waterfall）归入第一个匹配层级。纯语义（semantic-only）覆盖推理、沟通或输出风格；内容检测覆盖文件内容上的谓词；单事件覆盖单个命令、文件访问或网络连接；跨事件覆盖依赖跨操作时序或数据来源的策略。内容检测、单事件与跨事件的并集称为系统可观测（system-observable）。

1361 条策略中，只有 17% 是纯语义的。其余 83% 系统可观测，也就是内核级监测器原则上可以评估：38% 需要内容检测，29% 匹配单个 OS 事件，16% 需要跨事件状态。跨事件策略集中在开发流程领域，占全部跨事件策略的 39.5%。

![强制执行瀑布：纯语义、内容检测、单事件和跨事件在 1361 条策略上的分布](imgs/actplane-empirical_waterfall_enforcement.png)

跨事件策略反复出现四种模式。时序排序约束先后：「提交前先跑测试」要求一个事件发生在另一个之后，而不是更早某个时间点曾经发生过。跨文件一致性把不同工件上的变更绑在一起：「行为变了就同步改文档」把源码编辑与文档更新耦合起来。多步工作流执行带验证门的发布清单，每一步完成才能进入下一步。条件触发把操作耦在一起：「改了 spec 就必须同步 SDK」只在前置条件满足时触发。这些都无法从单个事件判定；强制执行必须记下跑过什么、以什么顺序、此后又发生了什么变化。此类策略并不少见：81% 的仓库至少有一条跨事件策略，43% 的仓库横跨全部四个强制执行档位。

## 项目与任务上下文让静态规则不够用

上下文依赖会把强制执行的难度再抬一层。1127 条系统可观测策略里，只有 26.4% 是自包含的。多数（64.2%）需要项目上下文：「测试套件」或「上游源码」必须对着具体仓库解析后，策略才能变成可执行规则。即便像 S5「不得修改上游源码」这样的单事件策略，也得先弄清哪些路径算「上游源码」，文件写入检查才能开火。另有 9.4% 需要任务上下文，例如「除非用户明确要求」或「未经批准不得操作」。

两种困难还会叠在一起。越是要跨事件追踪状态的策略，越少写明写规则所需的具体命令和路径。跨事件策略的上下文依赖高达 95%（77% 项目级，19% 任务级），内容检测策略则为 58%。「提交前跑测试」听起来简单，但强制执行引擎得知道该监视哪条测试命令、哪些源码目录算「相关编辑」，以及测试是通过了还是仅仅跑过。

![上下文瀑布：自包含、项目上下文和任务上下文在 1127 条系统可观测策略上的分布](imgs/actplane-empirical_waterfall_context.png)

一组固定的静态规则只能覆盖自包含的那部分。其余策略要先读仓库、解释当前任务，检查才有落点。

## 为什么现有层级执行不了这些规则

基于 prompt 的指令依赖模型自身遵从，既容易受 prompt 注入，又会在长上下文窗口里与用户任务 prompt 争夺注意力。独立的 agent 或 LLM 守卫可以在运行时检查 prompt、响应或行为轨迹，但这些检查本质上是概率性的。

工具调用守卫和应用级信息流控制系统在 harness 边界上能确定性拦截，却只能看到经 harness 中介的请求，看不到工具真正开始执行后的系统级效果。间接子进程、shell 外调或编译出来的二进制，都能绕过工具边界。agent 写一个含 `subprocess.run(["git", "push"])` 的 Python 脚本再执行，差距立刻显形：工具调用层看到的是「执行 python script.py」，而不是脚本里的 `git push`。

seccomp、AppArmor、Landlock、Tetragon 等 OS 机制管的是资源访问，不是开发者口中的那种「行为」。它们要求静态预写策略，报错也往往只剩一句令 agent 困惑的不透明 `EPERM`，既不解释违反了哪条规则，也不说明如何恢复。

这些层级合在一起，逼出一个联合要求。多数规则需要的项目或任务上下文住在 agent 一侧，所以 agent 自身必须能把策略转成具体规则；许多策略又定义了工具调用守卫看不见的事件顺序或数据流，所以规则必须具体到足以做确定性的 OS 级强制执行。弥合这道缝，正是 ActPlane 要处理的问题。

实证结果也因此落下两项设计要求。第一，策略规范必须既可由 agent 编写，又可被 OS 执行：agent 需要用最少的领域知识把自然语言策略落成具体规则，并借助语义反馈理解违规、完成恢复。第二，强制执行必须安全、隔离且高效：agent 编写的策略不能削弱更高权限设下的约束，不能干扰其他 agent 的策略，也不能拖慢 agent 的正常工作负载。

## 从指令文件到内核可执行的规则

![ActPlane 总览：离任务最近的 agent 编写具体策略 DSL，由内核编译并执行](imgs/actplane-illustration.png)

ActPlane 是对上述要求的一种回答：离任务最近的 agent 写具体策略，OS 内核负责执行。每条规则由五部分组成：标识治理对象的来源、目标操作（如 exec、write、connect）、效果、可选的时序门（temporal gate），以及用于语义反馈的原因字符串。论文的贯穿示例如下：

```
kill exec "git" "commit" unless after exec "go" "test" exits 0
```

这条规则会终止任何 `git commit`，除非 `go test` 在最近一次相关源码编辑之后成功退出过。这里省略的原因字段会在规则触发时向 agent 提供结构化解释。它正是研究中的 S6 模式，落到了具体仓库的命令与路径上。

三种效果对应指令与约束的区分。阻断（block）是操作前的同步拒绝，没有 TOCTOU 窗口：内核在系统调用执行前拦截，agent 可以改道。终止（kill）在操作开始后杀掉进程，防止 agent 在被阻断后切到别的通道。通知（notify）只传递引导、不阻止操作。约束用阻断或终止，指令用通知。

时序门让规则表达顺序，而不只是时间点谓词。`after ... since ...` 编码「一个事件必须发生在另一个之后」：测试必须在最近一次编辑之后跑过，而不是更早某个时间点跑过即可。`exits N` 区分成功与失败退出。谱系门（lineage gate）检查进程祖先，允许把操作限制在特定进程树里。

信息流标签（information-flow label）沿 fork、exec、read、write、connect 传播，并且是单调的：进程一旦读过带标签的对象，标签就不能去掉。进程读取 `.env` 时会获得该文件的来源标签；之后若尝试连接外部端点，匹配该标签的规则就会触发并阻断连接。研究中的 S7（「从 `.env` 读取的数据不得到达网络」）就是这样变成可执行跨事件规则的。

策略权限依赖时序信任边界。agent 启动前加载的规则属于高权限规则，对 agent 不可写。agent 及其子 agent 可以在子域中添加新规则或收窄现有规则，但不能削弱、移除或禁用继承的约束。运行时增量经 ring buffer 进入内核，由完全在内核中运行的权限检查器对照域层级校验，通过后才激活。信任计算基础由内核强制执行引擎与高权限策略组成，该边界以下的一切都是不可信执行。权限检查器在内核空间运行，被攻陷的用户空间 agent 无法在其域层级允许的范围之外改动活跃规则集。

标签单调也会带来过标记风险。长时间会话里，进程读得多了可能堆起过多标签，导致后续每个操作都触发规则。典型编码会话中，一个进程可能读几十个配置和源文件；没有缓解时，每次读取都加标签，读得足够多之后，每次写入或连接都会撞上某条规则。ActPlane 在生成新子进程时清除继承标签，把污染累积限制在单个进程的生命周期内，而不是整段会话。

607 条策略的数据集用到了 DSL 的多数特性，也验证了语言表达力。效果偏向观测：66% 的子句是通知，29% 是阻断，仅 5% 是终止，说明多数策略重在监控而非阻止。挂载点集中在代码执行（60% 为 exec）和文件变更（37% 为 write），网络与清理操作各不到 1%。跨事件特性使用广泛：28% 的策略使用 `after/since` 时序门，214 条使用 `unless` 编码例外。

实现规模也压得住。用户态编译器与运行器合计约 3.2K 行 Rust，eBPF 强制执行引擎约 1.8K 行 BPF C。BPF-LSM hook 处理操作前决策（阻断），tracepoint 处理观测与操作后终止（kill）。标签以 64 位掩码存在逐对象 BPF map 中，传播归结为一次按位 OR。引擎最多支持 128 条并发规则，数据集中观测到的最大仓库有 66 条策略。部署架构与机制细节见 [ActPlane：把 Agent Harness 强制执行下沉到内核 eBPF](https://eunomia.dev/zh/blog/2026/05/31/actplane-pushing-agent-harness-enforcement-down-to-kernel-ebpf/)。

## OS 级强制执行能否补上这道缝

策略翻译可以铺满整份数据。一个 Codex agent 在首次或第二次尝试中，就为全部 607 条 OS 可执行策略编译出 ActPlane 规则，607 条里只有 2 条需要语法重试。翻译用 7 个子 agent（其中 4 个并行）在 34 分钟内完成，把 607 条策略扩成 1283 条规则行。成本约每条 $0.028，消耗 1.7M 输入 token 和 177K 输出；按常见软件工程师费率估算，人工编写大约每条 $11。多数策略结构简单：74% 最多两个强制执行子句，第 95 百分位也在 152 token 以下。

带上下文的强制执行解决的违规，远多于任何基线。在决策合规基准（190 条轨迹、38 条源自实证研究的规则）上，ActPlane 的决策合规率（Decision Compliance Rate, DCR）达到 75.8%：

| 系统 | DCR |
|---|---|
| ActPlane | 75.8% |
| ActPlane-opaque（无反馈） | 53.7% |
| FIDES（工具级信息流控制） | 48.9% |
| Prompt-filter | 48.4% |
| Tool-regex | 45.3% |

基准为每条规则生成五条轨迹：两条合规轨迹用来压测误报（一条演练贴近边界的允许操作，一条执行技术上合规但外观像违规的操作）；三条违规轨迹（violation trace）反映部署中编码 agent 的执行路径类别。直接违规经工具调用触发被禁止操作；脚本违规把编写与执行拆到子进程；隐藏违规把副作用藏在中性入口后面的辅助工件里。

差距集中在违规轨迹上：ActPlane 在 114 条中正确解决 86 条，基线只有 27 到 44 条，提升 2.0 到 3.2 倍。所有系统在合规轨迹与直接违规上表现接近；ActPlane 的优势出现在脚本与隐藏轨迹上，因为被禁止的效果发生在子进程内或中性入口之后，工具调用拦截看不见。

语义反馈是合规与盲目重试的分水岭。完整 ActPlane 产生的正确违规轨迹结果，是无反馈引擎的三倍（86 对 27）。恢复率同样拉开：有语义反馈时 97.7%，无反馈时 31.4%。agent 知道操作为何被拒、需要修什么状态时，会改道完成任务；只收到一句空洞拒绝时，就会沿替代路径反复重试同一个被禁止操作。

开销放得进交互式开发。端到端开销在策略已加载、规则未命中的空命中配置下测量，覆盖两个工作负载。第一个是重放 68 个工具动作与 20 个 Bash 子进程的 agent 轨迹套件；第二个是 Linux 内核构建（`defconfig` + `vmlinux`，`make -j24`）。32 条活跃规则下，ActPlane 在 agent 轨迹上增加 1.9%，在内核构建上增加 6.5%；100 条规则时仍低于 8.4%。

微基准把单次系统调用开销的集中位置拆开。fork 与 exec 的绝对增加最大，跨规则数量范围为 3 到 69 微秒，相对 49 与 248 微秒的原生延迟仍算温和。文件与网络操作保持轻量：open 在 100 条规则时达到 13.4 微秒，write 与 connect 保持亚微秒级。整段工具调用系统调用序列上的 ActPlane 累积开销，比单次 2 到 10 秒的 LLM 推理轮次小五到六个数量级。策略更新也传得快：经用户空间 ring buffer 提交的单规则热重载，平均 26.3 微秒到达内核排空路径；立即的 exec 违规在 p50 176.4 微秒内被检测到，其中包含进程启动与事件传递。

这一排序在第二个模型上也站得住。DeepSeek-Pro V4 端到端复现保持系统排名不变，ActPlane 以 77.4% DCR 居首；两个模型设置之间的逐单元一致性给出 Cohen's kappa 0.822。

翻译质量同时拉动检测率与恢复率，因为规则过窄会漏掉违规，过宽会误伤合规操作。为衡量可改进性，每条假阴性轨迹的证据与纠正反馈被交给翻译 agent 修订一次。用修订后的规则重跑 28 条假阴性轨迹，恢复 26 条（93%），说明该 DSL 支持迭代完善。

真实编码任务在合成轨迹之外给出同样方向。在 OctoBench 的 21 个任务子集上（61 条 OS 可执行规则，覆盖 7 个仓库），相对无强制执行基线，ActPlane 把用户查询奖励提高 9.9 分，实现与测试奖励提高 9.7 分。收益超出纯合规检查，说明 OS 级强制执行加语义反馈，既帮 agent 守规则，也帮它更有效地完成任务。

外部安全基准则检验了论文自有数据集之外的覆盖面。在 361 个 OpenAgentSafety 个人助理任务中，ActPlane 以高权限规则预加载 agent 生成的安全策略，阻止了 74% 的基线不安全行为（106 起不安全结果中拦截 78 起）。28 起未拦截案例分为三类：聊天或语义伤害（不安全行为是没有 OS 可观测工件的消息）、不安全文件内容（不在 ActPlane 的主要覆盖范围内），以及服务端工件（效果是服务容器内的 WebDAV 上传或数据库变更，当前 hook 集观察不到）。

[ActPlane 源码](https://github.com/eunomia-bpf/ActPlane)已在 GitHub 开源。仓库中的 `policies/` 目录收录全部 64 个仓库的 607 条翻译规则，可直接当作编写自己指令文件的起点。

## 常见问题

### eBPF 足以解决 AI agent 安全问题吗？

eBPF 对文件写入、进程启动、网络连接等 OS 事件提供确定性强制执行，覆盖了 83% 的系统可观测策略。剩余 17% 的纯语义策略（推理、沟通风格或输出质量）落在内核之外，需要 harness 层控制。内容检测策略（需要读取文件内容的 38%）更适合交给代码检查器与静态分析器，不在 eBPF 的单事件与跨事件强制执行范围里。任务意图、策略权限、内容语义和隔离，仍要由内核强制执行点周围的各层承担。

### 行为基线能替代策略吗？

行为基线回答「这是否异常」，靠标记偏离历史模式的操作工作。策略回答「在当前任务下这是否被允许」。一次常规 `git commit` 按基线可以完全正常，却仍违反项目规则；一个陌生部署端点可以触发异常告警，但若用户明确要求，操作就合法。检测与授权是两件事，成熟系统会两者并用：异常信号可以提出候选规则，再经权限感知审查后变成真正执行的策略。

### AgentSight 会执行 ActPlane 的策略吗？

不会。[AgentSight](https://eunomia.dev/zh/blog/2025/08/26/agentsight-keeping-your-ai-agents-under-control-with-ebpf-powered-system-observability/) 是系统级分析与监测器，捕捉 agent 运行时行为。ActPlane 是强制执行引擎，拦截或重定向违反已加载策略的操作。二者互补：AgentSight 的运行时证据可以输入策略审查与修订，但观测与强制执行的职责保持分离。组织可以先用 AgentSight 发现 agent 频繁写入生产配置文件，再写对应的 ActPlane 规则阻止该模式，并附上禁止原因。

### ActPlane 仍然覆盖不了什么？

论文的威胁模型把边界写得很清楚。语义等价操作可以绕过基于字符串的匹配：agent 可以用自制 Git 客户端而不是调用 `exec git`，但底层的 `connect` 与 `write` 系统调用仍然可见。协议边界后的服务端效果（如服务容器内的 WebDAV 上传或数据库变更）也会逃出当前 hook 集。文件内容语义、内核攻陷、CAP_BPF 攻陷和侧通道均不在覆盖范围。17% 的纯语义策略（推理质量、沟通语气或输出格式）需要 harness 层处理，而不是内核强制执行。

数据集本身是这篇论文最独特的贡献。在 ActPlane 之前，没有公开度量把开发者要求 agent 遵守的规则，以及这些规则在强制执行需求上的分布，完整画出来。这些规则大多已经写在成千上万仓库的 CLAUDE.md 与 AGENTS.md 里；缺的是一层能读懂项目上下文、理解当前任务、把自然语言策略编译成具体内核级规则的强制执行能力。[ActPlane 仓库](https://github.com/eunomia-bpf/ActPlane)包含完整实现；把内核强制执行与隔离、身份和内容控制并置的三层安全模型，见[基于 eBPF 的不透明 AI Agent 运行时可观测与强制执行](https://eunomia.dev/zh/blog/2026/05/25/runtime-security-for-ai-agents/)。
