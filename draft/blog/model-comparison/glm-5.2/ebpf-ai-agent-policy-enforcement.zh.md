# 2116 条真实 AI Agent 规则需要 OS 强制执行和上下文

设想一个 AI coding agent 执行 `git commit`，内核看到的只是一个熟悉的进程在写熟悉的文件，没有任何异常。仓库里的 CLAUDE.md 写着"提交前必须跑完整测试套件"，而 agent 在上一次测试之后又改过源码。这条原话来自 [ActPlane 论文](https://arxiv.org/abs/2606.25189)的数据集，今天没有任何一层在强制执行它。

论文把整张图景压缩成一句话：64% 的语句是策略（policy），83% 涉及系统行为，74% 依赖无法静态预定义的上下文（context）。这三个数字背后，是第一次有人大规模度量开发者在 CLAUDE.md 和 AGENTS.md 里实际写下了什么、强制执行这些句子需要什么、以及一个可用的 OS 级引擎能不能补上这道缺口。本文梳理这次实证的发现、现有每一层强制执行为什么都触及不到这些规则，以及 ActPlane 的验证结果。

## 2116 条真实指令文件语句长什么样

[ActPlane](https://github.com/eunomia-bpf/ActPlane) 的出发点不是威胁模型，也不是攻击面，而是一个更朴素的问题：开发者已经告诉 agent 该做什么了，把这些指令变成可执行的规则到底需要什么？

语料覆盖 64 个含 CLAUDE.md 或 AGENTS.md 的热门仓库（中位数 20K GitHub star，快照 2026-05-23），合计 84 份指令文件、2116 条独立语句。早期关于 agent 指令文件的研究在文件或章节标题粒度做分类，把行为规则埋在了长长的架构描述里。ActPlane 改成对每条语句单独打标，六行的目录说明和一行"不得直接推到 main"不再被算作同一个分析单元。

按这种粒度提取语句需要一条谨慎的流水线。两遍 LLM agent 辅助流程为每条语句记录源行范围和四个标签（内容类型、主题、强制执行层级、上下文需求）。验证脚本检查源覆盖完整性以及对原始文件的逐字 span 匹配，再由两个独立 agent（Claude 和 Codex）交叉核对标签。最后 100 条分层抽样语句经过人工标注者独立审核，确认标签正确。

按语句计数和按行计数会得到完全不同的画面。2116 条语句中 64% 是策略，它们要求、禁止或约束某个具体 agent 行为；其余 36% 是描述性上下文，例如架构说明或项目背景。各仓库的策略密度从 0% 到 97% 不等，70.1% 的仓库策略数量多于描述。如果改按行数统计，结果只剩 49% 的策略，因为描述平均 6.8 行而策略平均 3.6 行。这个差距正是语句级分析值得做的原因。

12 个改编自先前指令文件研究的主题类别（应用于语句粒度）揭示了策略集中在哪里。开发流程和实现细节两类的策略占比最高，分别达到 87% 和 85%；架构以描述性内容为主，策略仅占 23%，因为目录布局和设计摘要构成了这些章节的主体。

![64 个含 CLAUDE.md 或 AGENTS.md 的仓库中每个仓库的策略比例](imgs/actplane-empirical_rq1_policy_fraction.png)

数据集中的五条真实语句让人直观感受强制执行要处理的差异幅度：

| 语句 | 强制执行层级 | 上下文 |
|---|---|---|
| S4: "Never push to main directly." | 单事件（per-event） | 自包含（self-contained） |
| S5: "Never modify upstream source code." | 单事件 | 项目级 |
| S6: "Run the full test suite before committing." | 跨事件（cross-event） | 项目级 |
| S7: "Data read from .env must not reach the network." | 跨事件 | 项目级 |
| S8: "Do not update dependencies without approval." | 单事件 | 任务级 |

## 83% 的策略系统可观测，16% 是跨事件

论文为每条策略找到强制执行瀑布（enforcement waterfall）中第一个它能满足的层级。纯语义（semantic-only）涵盖推理、沟通或输出风格；内容检测（content inspection）涵盖文件内容上的谓词；单事件涵盖单个命令、文件访问或网络连接；跨事件涵盖依赖跨操作时间顺序或数据来源的策略。内容检测、单事件、跨事件三层并集叫做系统可观测（system-observable），因为内核级监测器原则上就能评估。

1361 条策略中 83% 落在系统可观测区域，其中 38% 需要内容检测，29% 匹配单个 OS 事件，16% 需要跨事件状态。其余 17% 是纯语义的。跨事件策略集中在开发流程主题里，仅这一类就占全部跨事件策略的 39.5%。

![强制执行瀑布：纯语义、内容检测、单事件和跨事件在 1361 条策略上的分布](imgs/actplane-empirical_waterfall_enforcement.png)

跨事件策略呈现四种反复出现的形状。时序排序约束事件先后，"提交前先跑测试"要求一个事件发生在另一个之后，而不是仅仅在更早的某个时间点发生过。跨文件一致性把跨工件的变更耦合起来，"行为变更时同步更新文档"就把源码编辑和文档更新绑在一起。多步工作流执行带有验证门的发布清单，每一步必须完成才能进入下一步。条件触发耦合操作，"改了 spec 就同步更新 SDK"只在前置条件满足时才触发。这些策略都无法从单个事件判断，强制执行引擎必须记录运行了什么、以什么顺序、以及从那时起发生了什么变化。它们分布也很广，81% 的仓库至少包含一条跨事件策略，43% 的仓库横跨全部四个强制执行层级。

上下文依赖让问题进一步叠加。1127 条系统可观测策略中只有 26.4% 是自包含的。64.2% 需要项目上下文，因为"测试套件"或"上游源码"这类措辞必须对着具体仓库解析后才能变成可执行规则。即便是 S5 这种单事件策略（"不得修改上游源码"），也要先解析哪些路径构成"上游源码"，文件写入检查才能生效。另有 9.4% 需要任务上下文，例如"除非用户明确要求"或"未经批准不得操作"。

两种困难恰好落在同一批策略上。跨事件策略的上下文依赖高达 95%（77% 项目级，19% 任务级），而内容检测策略只有 58%。"提交前跑测试"听起来简单，但强制执行引擎需要知道监视哪条测试命令、哪些源码目录算"相关编辑"、以及测试是通过了还是仅仅运行过。

![上下文瀑布：自包含、项目上下文和任务上下文在 1127 条系统可观测策略上的分布](imgs/actplane-empirical_waterfall_context.png)

固定的静态规则只能覆盖自包含的那一小片。其余策略都要求引擎先读取仓库、解释当前任务，然后任何检查才能开始跑。

## 现有的 prompt、工具、沙箱三层为什么都触及不到这些规则

现有三层针对语料里的规则全都失效，每一层失效的原因还各不相同。

写在 prompt 里的指令依赖模型自身的遵从能力，但它容易受 prompt 注入攻击，并且会随着对话变长而失重。CLAUDE.md 里的一条规则要在长上下文窗口中与用户的任务 prompt 争夺注意力，间接违反尤其容易。被告知"不要删除文件"的 agent 可以写一个包含 `rm -rf` 的 Makefile target，然后调用 `make clean`，技术上从未违反指令。

工具调用层面的守卫和应用级信息流控制系统在 harness 边界确定性地拦截，但它们只能观察经过 harness 中介的请求。间接子进程、shell 外调、编译出来的二进制都能绕过工具边界。设想一个 agent 写了包含 `subprocess.run(["git", "push"])` 的 Python 脚本再执行它，工具调用层看到的是"执行 python script.py"，看不到脚本里的 `git push`。

像 seccomp、AppArmor、Landlock、Tetragon 这样的 OS 机制控制的是资源访问，而不是开发者真正描述的行为。它们要求静态预写策略，检查失败时报错也很不透明，一句 `EPERM` 既不说明违反了哪条规则，也不告诉 agent 如何恢复，结果就是 agent 通过替代路径反复重试同一个被禁操作。

这三层失效指向同一个结论。大多数规则需要存在于 agent 处的项目或任务上下文，因此 agent 自身必须能把策略转化成具体规则；而这些策略中有许多定义了事件顺序或数据流，对工具调用层的守卫又不可见，因此规则必须具体到能让 OS 级强制执行确定性地工作。论文把这个结论变成两项设计要求。策略规范必须 agent 可写且 OS 可执行，并配以语义反馈（semantic feedback）让 agent 理解违规并恢复；强制执行必须安全、隔离且高效，agent 编写的策略不能削弱更高权限的约束，不能影响其他 agent，也不能拖慢工作负载。

## ActPlane 如何验证这些要求

ActPlane 是论文为了检验这些要求能不能扛住真实规则而构建的引擎。完整的机制设计放在[姊妹篇：把 Agent Harness 强制执行下沉到内核 eBPF](https://eunomia.dev/zh/blog/2026/05/31/actplane-pushing-agent-harness-enforcement-down-to-kernel-ebpf/)里，本节只展开实证研究用到的那部分。

![ActPlane 总览：离任务最近的 agent 编写具体策略 DSL，由内核编译并执行](imgs/actplane-illustration.png)

每条规则由五个部分组成。来源声明标识治理对象，目标操作（如 exec、write、connect）命名动作，效果设定后果，可选的时序门（temporal gate）编码顺序，原因字符串承载语义反馈。论文的贯穿示例可以把它具体化：

```
kill exec "git" "commit" unless after exec "go" "test" exits 0
```

这条规则会终止任何 `git commit`，除非 `go test` 在最近一次相关源码编辑之后成功退出过。这里省略的原因字段会在规则触发时给 agent 一份结构化的解释。

三种效果对应了指令与约束的区分。阻断（block）是操作前的同步拒绝，没有 TOCTOU 窗口，内核在系统调用执行之前就拦截它，agent 可以改道重试。终止（kill）在操作开始后杀掉进程，防止 agent 被阻断后切换到其他通道。通知（notify）只传递引导信息而不阻止操作。约束使用阻断或终止，指令使用通知。

时序门让规则表达顺序而不仅是时间点谓词。`after ... since ...` 编码的是一个事件必须发生在另一个之后，测试必须在最近一次编辑之后跑过，而不是在更早的某个时间点跑过就行。`exits N` 限定符区分成功退出和失败退出。谱系门（lineage gate）检查进程祖先链，让规则可以把操作限制在特定进程树里。

信息流标签（information-flow label）沿 fork、exec、read、write、connect 传播，且这种传播是单调的。进程一旦读取了带标签的对象，标签就再也去不掉。当进程读取 `.env` 时，它获得该文件的来源标签；如果它之后尝试连接外部端点，匹配该标签的规则就会触发并阻断连接。数据集中的 S7，"从 `.env` 读取的数据不得到达网络"，就是这样变成可执行的跨事件规则的。

策略权限建立在时序信任边界上。Agent 启动前加载的规则是高权限规则，对 agent 不可写。Agent 及其子 agent 可以在子域中添加新规则或收窄继承来的约束，但不能削弱、移除或禁用上层设定的约束。运行时增量通过 ring buffer 到达内核，经过完全在内核中运行的权限检查器核对域层级后才会激活。因为检查器完全跑在内核空间，被攻陷的用户态 agent 也无法越过自身域层级去修改活跃规则集。

长时间运行的会话会因为标签单调而面临过标记风险。在一次典型的编码会话里，一个进程可能读取几十个配置和源文件，若不缓解，每次读取都会累加一个标签，最终每次写入或连接都会匹配某条规则。ActPlane 在生成新子进程时清除继承来的标签，把污染累积限制在每个进程的生命周期内，而不是整个会话。

607 条策略的数据集运用了 DSL 的大部分特性，证实这门语言能覆盖真实指令文件里的写法。效果偏向观测，66% 的子句是通知，29% 是阻断，只有 5% 是终止，这与"大多数策略是监控而非阻止"的直觉吻合。挂载点集中在代码执行（60% exec）和文件变更（37% write），网络和清理操作各不到 1%。跨事件特性被广泛使用，28% 的策略用了 `after/since` 时序门，214 条用 `unless` 编码例外。

实现规模之所以能保持紧凑，是因为规则语言让内核要做的工作足够简单。用户态编译器和运行器约 3.2K 行 Rust，eBPF 强制执行引擎约 1.8K 行 BPF C。BPF-LSM hook 处理操作前决策（阻断），tracepoint 处理观测和操作后终止。标签以 64 位掩码存在逐对象的 BPF map 里，传播归结为一次按位 OR，这也是引擎能在 128 条并发规则下从容应对数据集中最大的仓库（66 条策略）的原因。

## 检测、恢复率与开销：真实策略上的评估结果

策略翻译覆盖了整个数据集。一个 Codex agent 在首次或第二次尝试中就把全部 607 条 OS 可执行策略编译为 ActPlane 规则，607 条里只有 2 条因语法错误重试。整轮在 34 分钟内跑完，用到 7 个子 agent（4 个并行），把 607 条策略扩展为 1283 行规则，成本约每条策略 $0.028，消耗 1.7M 输入 token 和 177K 输出。与人工编写约每条 $11 的水平相比，自动翻译便宜到可以对每个仓库都跑一遍。大多数策略结构简单，74% 最多两个强制执行子句，第 95 百分位也在 152 token 以下。

在决策合规基准（190 条轨迹、38 条源自实证研究的规则）上，带上下文的强制执行比任何基线都多解决一大截违规，ActPlane 的决策合规率（Decision Compliance Rate, DCR）为 75.8%：

| 系统 | DCR |
|---|---|
| ActPlane | 75.8% |
| ActPlane-opaque（无反馈） | 53.7% |
| FIDES（工具级信息流控制） | 48.9% |
| Prompt-filter | 48.4% |
| Tool-regex | 45.3% |

基准为每条规则生成五条轨迹。两条合规轨迹用于压力测试误报，一条演练接近边界的允许操作，另一条执行技术上合规但形似违规的操作。三条违规轨迹反映部署中编码 agent 的执行路径类别。直接违规通过工具调用触发被禁操作；脚本违规把编写和执行拆分到子进程；隐藏违规把副作用嵌进中性入口点后面的辅助工件里。

ActPlane 的优势在违规轨迹上拉开。它正确解决 114 条中的 86 条，基线只有 27 到 44 条，提升了 2.0 到 3.2 倍。所有系统在合规轨迹和直接违规上表现相当，差距出现在脚本和隐藏两类轨迹上，因为被禁止的效果发生在子进程内或中性入口点后面，工具调用拦截根本观察不到。

语义反馈是合规与盲目重试的分水岭。完整 ActPlane 产生的正确违规轨迹结果是无反馈引擎的三倍，86 对 27，恢复率也呈同一规律，有反馈时 97.7%，无反馈时 31.4%。一旦 agent 知道操作为何被拒、需要修复什么状态，它就会改道完成任务；只收到一句空洞的拒绝时，它会通过替代路径反复重试同一个被禁操作。

端到端开销落在开发工作流可以接受的区间。论文在策略已加载但不触发任何规则的配置下测了两个工作负载。一个是重放 68 个工具动作和 20 个 Bash 子进程的 agent 轨迹套件，另一个是 Linux 内核构建（`defconfig` + `vmlinux`，`make -j24`）。32 条活跃规则下，ActPlane 在 agent 轨迹上加 1.9%，在内核构建上加 6.5%；100 条规则时仍低于 8.4%。

微基准测试定位了单次系统调用开销集中在哪里。fork 和 exec 的绝对增量最大，跨规则数量为 3 到 69 微秒，但相对于它们 49 和 248 微秒的原生延迟仍然适度。文件和网络操作保持轻量，open 在 100 条规则时达到 13.4 微秒，write 和 connect 保持亚微秒级。整个工具调用系统调用序列累积的 ActPlane 开销，比一次 2 到 10 秒的 LLM 推理小五到六个数量级，所以 harness 的开销实际上消失在 agent 循环里。策略更新传播迅速，通过用户态 ring buffer 提交的单规则热重载平均 26.3 微秒到达内核排空路径，立即的 exec 违规在 p50 176.4 微秒内被检测到（含进程启动和事件传递）。

系统排名在第二个模型上仍然成立。DeepSeek-Pro V4 端到端复现保持排序不变，ActPlane 以 77.4% DCR 居首，两个模型设置之间的逐单元一致性给出 Cohen's kappa 0.822。

翻译质量同时决定检测和恢复，因为规则过窄会漏掉违规，过宽会匹配合规操作。为衡量可改进性，论文把每条假阴性轨迹的证据和纠正反馈喂给翻译 agent，允许它修订一次规则。用修订后的规则重跑 28 条假阴性轨迹，恢复 26 条（93%），说明 DSL 支持迭代完善。

真实编码任务在合成轨迹之外印证了同一规律。在 OctoBench 的 21 个任务子集上（61 条 OS 可执行规则，覆盖 7 个仓库），ActPlane 的用户查询奖励比无强制执行基线高 9.9 分，实现与测试奖励高 9.7 分。收益超出了合规类型检查的范围，说明 OS 级强制执行加语义反馈不仅帮助 agent 遵守规则，也帮助它们更有效地完成任务。

外部安全基准验证了论文自身数据集之外的广度。在 361 个 OpenAgentSafety 个人助理任务上，ActPlane 在 agent 启动前以高权限规则预加载 agent 生成的安全策略，阻止了 74% 的基线不安全行为（106 起不安全结果中拦截 78 起）。28 起未拦截案例归为三类：聊天或语义伤害，不安全行为是一条没有 OS 可观测工件的消息；不安全文件内容，不在 ActPlane 的主要覆盖范围内；服务端工件，效果是服务容器内的 WebDAV 上传或数据库变更，当前 hook 集观察不到。

[ActPlane 源码](https://github.com/eunomia-bpf/ActPlane)已在 GitHub 开源。`policies/` 目录包含全部 64 个仓库的 607 条翻译规则，可以作为编写自己指令文件的起点。

## 常见问题

### eBPF 足以解决 AI agent 安全问题吗？

eBPF 对 OS 事件（文件写入、进程启动、网络连接等系统行为）提供确定性强制执行，覆盖了 83% 的系统可观测策略。剩余 17% 涵盖推理、沟通风格或输出质量，需要内核之外的 harness 层控制。内容检测策略，即需要读取文件内容的那 38%，更适合由代码检查器和静态分析器处理，不在 eBPF 单事件和跨事件强制执行的范围里。任务意图、策略权限、内容语义和隔离，都需要内核强制执行点周围的层来承担。

### 行为基线能替代策略吗？

行为基线回答"这是否异常"，靠标记偏离历史模式的操作工作。策略回答的是"在当前任务下这是否被允许"。一次常规 `git commit` 按基线标准可以完全正常，同时违反一条项目规则；一个陌生的部署端点可以触发异常告警，但如果用户明确要求过，它就是合法的。检测和授权是两个不同的问题，成熟系统会同时使用两者。异常信号可以提议候选规则，再经过权限感知审查后变为已执行的策略。

### AgentSight 会执行 ActPlane 的策略吗？

不会。[AgentSight](https://eunomia.dev/zh/blog/2025/08/26/agentsight-keeping-your-ai-agents-under-control-with-ebpf-powered-system-observability/) 是系统级分析器和监测器，捕捉 agent 在运行时的行为。ActPlane 是强制执行引擎，拦截或重定向违反已加载策略的操作。两者是互补角色，AgentSight 的运行时证据可以为策略审查和优化提供输入，而观测和强制执行的职责保持分离。一个组织可以先用 AgentSight 发现 agent 频繁写入生产配置文件，再编写对应的 ActPlane 规则来拦截这种模式并附上禁止原因。

### ActPlane 仍然覆盖不了什么？

论文的威胁模型明确划出边界。语义等价操作可以绕过基于字符串的匹配，agent 可以用自制 Git 客户端而非调用 `exec git`，但底层的 `connect` 和 `write` 系统调用仍然可见。协议边界后面的服务端效果，如服务容器内的 WebDAV 上传或数据库变更，也逃脱了当前 hook 集。文件内容语义、内核攻陷、CAP_BPF 攻陷和侧通道均不在覆盖范围内。17% 的纯语义策略，涵盖推理质量、沟通风格或输出格式，需要 harness 层而非内核强制执行来处理。

## 有了这个数据集，下一步会变什么

在 ActPlane 之前，没有人度量过开发者到底要求 agent 遵守什么规则，也没有人分析过这些规则在强制执行需求上的分布。规则本身早就写在成千上万仓库的 CLAUDE.md 和 AGENTS.md 里，缺的是一个能读懂项目上下文、理解当前任务、把自然语言策略编译成具体内核级规则的强制执行层。[ActPlane 仓库](https://github.com/eunomia-bpf/ActPlane)包含完整实现和全部规则集，将内核强制执行与隔离、身份和内容控制并置的三层安全模型见[基于 eBPF 的不透明 AI Agent 运行时可观测与强制执行](https://eunomia.dev/zh/blog/2026/05/25/runtime-security-for-ai-agents/)。
