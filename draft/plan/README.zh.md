# 每日内容产出计划说明

> 状态：内部执行草案，不发布。创建日期：2026-07-19。

本目录按月份保存每日运营计划。`eunomia-content-patrol` 是执行和维护
这类计划的 skill，并在每次运行时读取本说明。当前阶段默认先清理已有
内容的跨平台同步积压，再安排新内容；知乎和掘金优先。

## 怎么写每日计划

每个日期使用简短 checklist，不用表格。至少包含：

- `[ ] 怎么做：` 写清源文件、目标平台和执行顺序。
- `[ ] 主任务：` 当天只设一个主要同步对象；可附一个状态检查。
- `[ ] 已完成：` 执行后勾选，补文件、平台 URL 或 ledger 结果。
- `[ ] 未完成/阻塞：` 执行后写尚未完成的具体原因；没有则写“无”。
- `[ ] 下一步：` 写明第二天可直接开始的文件或平台动作。

计划项要能直接执行，不写“优化内容”“做运营”之类空任务。已有匹配
语言的长文同步时，标题和正文保持原样，只做格式、图片和平台元数据
处理。新 tutorial 合入后，检查中英文源、页面可用性和 ledger，在当前月
或下月最近的空档安排知乎/掘金同步。

## 执行顺序

1. 打开当前月份文件，例如 `draft/plan/2026-07.zh.md`。
2. 找到当天日期下面的 list。
3. 确认昨天是否留下阻塞或待确认项。
4. 优先完成计划内尚未同步的知乎/掘金长文；没有可同步项时再做新稿、雷达或转发候选。
5. 如果涉及平台真实操作，只用侧边栏/应用内浏览器；没有当前任务的明确发布授权时停在预览或确认页。
6. 直接回填当天的“已完成、未完成/阻塞、下一步”，并在需要详细证据时更新 `draft/content-daily-log-YYYY-MM.md`。
7. 如确认了平台状态，更新 `.github/publisher/media/platforms/*.json`、`published.md` 或 `not-published.md`。

## 每日完成标准

- 至少完成一个有意义的动作：一个现有长文进入平台预览/发布/确认状态、ledger 状态确认，或有证据的明确跳过理由。
- 不把 daily log、空草稿、形式化截图当作当天核心产出。
- 当天 checklist 已回填怎么做、做了什么、没做什么和下一步。
- 同步积压未清完时，不用新的选题或短帖替代当天主任务。
