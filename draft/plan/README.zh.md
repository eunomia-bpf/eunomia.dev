# 每日内容产出计划说明

> 状态：内部执行草案，不发布。创建日期：2026-07-19。

本目录按月份保存每日运营计划。`eunomia-content-patrol` 是执行和维护
这类计划的 skill，并在每次运行时读取本说明。当前阶段默认先清理已有
内容的跨平台同步积压，再安排新内容；知乎和掘金优先。

## 怎么写每日计划

每个日期使用简短 checklist，不用表格。至少包含：

- `怎么做：` 写清源文件、本地 artifact、目标平台和执行顺序。
- `主任务：` 当天只设一个主要同步对象，可以流水线处理它的其他平台版本。
- 每同步到一个平台，单列一个 `[ ] 平台：` 任务点。同一篇同步到知乎和掘金算两个任务点，不能合并成“知乎/掘金”一个 checkbox。
- `已完成：` 执行后把对应平台改为 `[x]`，紧跟平台 URL、审核状态和 ledger 结果。
- `下一步：` 写明第二天可直接开始的源文件或平台动作。

阻塞项只写在每个月文档开头的“本月阻塞与排除”中，不在每天重复。
阻塞解除后先更新月首状态，再把平台任务排进具体日期。每个月都要覆盖
所有当前可执行的平台；明确暂停、没有账号或与内容不匹配的平台，也在
月首写清原因，不为了凑矩阵强行发布。

计划项要能直接执行，不写“优化内容”“做运营”之类空任务。已有匹配
语言的长文同步时，标题和正文保持原样，只做格式、图片和平台元数据
处理。新 tutorial 合入后，检查中英文源、页面可用性和 ledger，在当前月
或下月最近的空档安排知乎/掘金同步。

## 排期前先做完

盘点历史发布、核对平台 URL、确认审核状态、检查是否重复，这些是写计划
前的准备工作，能在当前任务完成就立即完成，不能占用未来日期。未来每日
项只安排明确的发布、已发内容检查或有主题的报告。每个平台维护独立
时间线，隔日是默认防刷屏参考，不是硬门槛；一天可以同步多个平台，
同一平台也可以为完成一批相关同步连续发布几天。近期新增和早期基础
内容交替，避免在较长周期内密集灌入历史文章。Reddit 等社区平台按
社区匹配度发布，不机械凑固定频率。

当前同步顺序：

- 掘金优先补真实缺失的中文 tutorial，穿插重要中文 blog。
- 知乎只补完整可见历史核对后确认缺失的中文内容。
- Medium/DEV 只补核对后缺失的英文长文。
- X/LinkedIn 为已发布长文写简短平台原生帖子和链接。
- Reddit 等社区平台只在当前讨论和社区规则匹配时发布。

## 执行顺序

1. 打开当前月份文件，例如 `draft/plan/2026-07.zh.md`。
2. 找到当天日期下面的 list。
3. 先检查月首阻塞清单是否有变化，再看昨天的完成项和下一步。
4. 优先完成计划内尚未同步的知乎/掘金长文；没有可同步项时再做新稿、雷达或转发候选。
5. 如果涉及平台真实操作，只用侧边栏/应用内浏览器；没有当前任务的明确发布授权时停在预览或确认页。
6. 逐个平台回填 URL、审核和 ledger；不把一个平台的完成状态外推到另一个平台。当天确实需要运行记录时，写入 `draft/media/YYYY-MM-DD/run-log.md`。
7. 如确认了平台状态，更新 `.github/publisher/media/platforms/*.json`、`published.md` 或 `not-published.md`。

## 每日完成标准

- 至少完成一个计划中的平台发布任务点；无发布日则完成计划写明的发布后检查或主题报告。历史“发没发”核账在排期前完成，不作为未来日任务。
- 不创建月度 daily log；当天确有异常、反馈或待跟进事项时，run log 只放在对应日期的 `draft/media/` 目录，也不能替代当天核心产出。
- 不创建 figure inventory、platform hooks、单篇 publish QA 或其他仅用于证明流程执行过的一次性文件。
- 当天 checklist 已按平台回填完成状态、URL 和下一步。
- 同步积压未清完时，不用新的选题或短帖替代当天主任务。
