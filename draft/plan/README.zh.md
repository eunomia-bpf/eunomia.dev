# 内容发布计划说明

> 状态：内部执行草案，不发布。更新时间：2026-07-23。

本目录只维护一个滚动发布队列：
`draft/plan/publishing-queue.zh.md`。发布顺序会变化，内容就绪、平台审核、
上一项反馈和用户确认都可能改变下一步，因此不再把候选内容提前塞进具体
日期，也不再维护月度日历式计划。

## 怎么写队列

- 从上到下按最早可执行顺序排列。
- 每个平台同步单列一个 checkbox；同一篇同步到两个平台算两个任务点。
- 每项写清状态、平台、源文件和必要检查，不创建“准备一天”“检查一天”
  之类空占位。
- 状态只使用 `可发布`、`待确认`、`待复核`、`候补`、`阻塞`。
- 只有 `可发布` 是定时巡检的真实发布授权；其余状态只能准备、检查或等待。
- 完成后改为 `[x]`，回填公开 URL、审核状态和 ledger 结果。

## 时间规则

- 每天最多向一个公开平台发布一篇内容；eunomia.dev、知乎、掘金、
  Medium、DEV、X、LinkedIn 和 Reddit 都计入上限。
- 完成一项后，下一项最早在下一个自然日进入发布窗口。平台需要更长
  间隔、前一项仍在审核或内容尚未就绪时，队列自然顺延。
- 暂停和阻塞写在队列顶部。已经发生的发布事实保存在 platform ledger
  和同日 `draft/media/YYYY-MM-DD/run-log.md`，不复制成日历流水账。
- Weekly Analysis 使用“距离上一篇至少 7 天”的条件，不预占未来日期；
  来源和 thesis 达不到门槛时跳过。

## 排队前先做完

历史发布核对、重复检查、源文件质量修复和账号状态确认应尽可能立即完成，
不能伪装成未来任务。长文同步保持源标题和正文，只处理格式、图片和平台
元数据；源内容需要修正时先更新源文件，再进入队列。

当前同步原则：

- 知乎和掘金优先处理 ledger 确认缺失的中文长文和 tutorial。
- Medium 和 DEV 只处理 ledger 确认缺失的英文长文。
- X 和 LinkedIn 默认分享已发布长文链接，外加 1-2 句核心钩子；项目演示
  可以使用简短原生帖子和真实图片、GIF 或视频。
- Reddit 等社区平台只在内容与社区规则、当前讨论明确匹配时发布。

## 执行

1. 读取 `publishing-queue.zh.md` 顶部的暂停和阻塞。
2. 找到最靠上的、条件满足且标记为 `可发布` 的任务；没有这类任务时不
   推断发布授权。
3. 需要平台操作时，调用对应 publisher skill，并只使用普通可见浏览器 UI。
4. 发布前使用本地完成的 platform artifact；发布后检查公开页并更新 ledger。
5. 平台文案和上传 artifact 放在
   `draft/media/YYYY-MM-DD/<source-slug>/`。只有真实异常、反馈或待跟进
   事项才写同日 `run-log.md`。

不创建月度 daily log、figure inventory、platform hooks、单篇 publish QA
或其他仅用于证明流程执行过的一次性文件。
