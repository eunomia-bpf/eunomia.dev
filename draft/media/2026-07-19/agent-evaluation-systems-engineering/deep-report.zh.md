# Agent 评测正在变成系统工程：为什么一个分数无法代表可部署的 Agent

过去两年，Agent 评测最常见的问题是“哪个模型得分更高”。到了 2026 年，这个问题正在失去一部分解释力。一个 Agent 的实际行为不仅来自模型，还来自系统指令、可用工具、权限范围、记忆、执行环境和失败恢复机制。模型不变，只要其中一个组件发生变化，成功率、成本和风险就可能随之改变。

近期出现的一组研究和公共测试设施，把这种变化呈现得格外清楚。它们并没有共同使用一套术语，也没有给出完全一致的解决方案，但都在重新定义评测对象：不再只测模型能不能回答问题，而是测一个配置明确的系统，能否在具体环境中完成长任务、遵守约束，并在条件变化时保持可靠。

**核心判断是：Agent 评测正在从模型能力测试演变为部署配置的系统实验。** 单一分数仍然有比较价值，但不能单独回答一个 Agent 是否适合上线。工程上真正有用的证据，至少还要说明测了什么配置、在哪种环境中测、失败从哪里开始，以及结果能否被持续复现。

## 结论摘要

第一，Agent 的表现不是模型单独产生的。2026 年几项大规模研究已经能够在相同模型下，仅改变 Agent 架构、工具暴露方式或控制组件，就观察到显著的成功率和成本差异。所谓“某模型的 Agent 能力”，如果没有同时说明 scaffold、工具、权限和评测接口，通常是不完整的说法。

第二，最终成功率会丢失最需要修复的信息。两个系统可能得到相同分数，一个因为过早终止失败，另一个因为缺少证据核验失败；一个虽然完成任务，却使用了不安全的操作路径。它们需要完全不同的工程处置。轨迹、最终环境状态、失败恢复和副作用因此不是附属日志，而是评测结果的一部分。

第三，评测越接近生产环境，真实性、可复现性和归因能力之间的冲突越强。固定容器和静态任务便于重复，却可能省略真实环境中的状态漂移、权限差异和接口故障；直接使用生产协议更贴近部署，却增加外部依赖和时间变化。没有单一 benchmark 能同时最大化这些目标。

第四，对高风险 Agent，最有价值的问题不是“平均能做多少”，而是“在哪个条件下开始以什么方式失败”。这要求评测从固定题集扩展到配置迁移、扰动实验、极限发现和事故回放，并把可检测性、可停止性和恢复能力纳入结果。

第五，成熟的 Agent 评测不会由一个万能排行榜替代所有测试。更可行的形态是分层证据：组件契约测试负责快速定位，冻结场景负责稳定回归，全系统轨迹测试负责验证长流程，配置迁移负责检查变化，极限发现和生产回放负责暴露边界。公开排行榜仍有价值，但它只能占其中一层。

## 一个看似矛盾的起点

2026 年 7 月 15 日发布的一项临床安全审计研究，为理解这个变化提供了一个很好的切口。研究者要求 Agent 在 Docker 容器中，仅通过 bash 接口，根据书面规范实现四类攻击，计算临床模型的 Security Posture Score，并生成结构化 JSON 报告。Claude Sonnet 4.6 和 GPT-4.1 在各自 18 次运行中都得到满分，GPT-4o 则只完成了 61% 的运行。失败并不都来自不会实现攻击，其中包括提前结束会话、手工聚合算错，以及创建了空提交文件。[论文及开放任务](https://arxiv.org/abs/2607.13411)

这项实验一共运行 54 次，覆盖三个模型、六个任务变体和每个变体三次重复。任务不是让模型口头解释安全审计，而是要求它在隔离环境中读取数据、编写和执行攻击代码、汇总指标、生成符合 schema 的最终文件。评分器同时检查任务完成情况与产物内容。也就是说，它测到的是模型、Agent 执行循环、bash 工具、容器、任务规范和评分脚本共同构成的系统。

这一区分很重要。GPT-4o 的失败横跨至少三层：提前结束属于控制流或停止条件问题，算术聚合错误属于任务推理或实现问题，空文件属于产物提交问题。若只保留 61% 这个数字，团队无法判断应该更换模型、增加结果校验、修复终止策略，还是在提交前加入文件完整性检查。一个分数完成了排序，却没有完成诊断。

如果只看摘要，这很容易被解释成“Agent 已经能够自主完成临床安全审计”。论文自己的限制部分却给出了更谨慎的边界：公开数据集较小且干净，任务直接提供攻击伪代码，三个模型和每个变体三次运行不足以稳定估计方差，评分容差在接近零的指标上可能过于宽松，六个任务变体也不能直接代表真实医院中的复杂深度学习系统。

这些限制对应不同类型的有效性问题。提供攻击公式和伪代码，意味着实验主要验证实现与执行能力，不验证 Agent 能否自行选择正确的审计方法。干净的小型数据集降低了数据缺失、格式变化和资源压力。MIMIC-IV 的访问限制又使部分复现依赖额外授权。三次重复可以发现明显不稳定，却很难刻画低频失败。研究覆盖的攻击也不包含数据投毒、模型抽取和迁移攻击，因此不能外推为完整的临床 AI 安全能力。

这不是论文自相矛盾，而是在说明评测分数的正确含义。满分证明的是：在给定模型、指令、工具接口、数据和评分器的条件下，两个 Agent 能够稳定执行这项被明确规定的工作。它没有证明相同 Agent 面对缺失数据、变化中的工具、不同权限或未给出攻击方法的真实部署时仍然可靠。

换句话说，评测结果不是 Agent 的固有属性，而是 Agent 与环境共同产生的条件性结果。

## 评测对象是怎样改变的

传统语言模型评测通常把输入固定为题目，把输出固定为文本，再用标准答案或偏好判断质量。即使模型存在随机性，实验对象仍相对清晰：模型版本、提示和解码参数决定一次生成。Agent 把这个闭环打开了。它会读取外部状态、选择工具、产生副作用、接收反馈，并根据中间结果改变后续计划。

因此，Agent 评测至少经历了四个对象层级。第一层是模型输出，关注单轮答案是否正确。第二层是工具使用者，关注模型能否选择正确工具并构造合法参数。第三层是有状态执行系统，关注多轮计划、记忆、恢复、停止条件和最终环境状态。第四层是部署配置，进一步包含权限、网络、依赖版本、策略层、成本限制、用户行为和外部服务变化。

层级越高，底层能力仍然重要，但已经不足以解释观察到的行为。一个模型可能知道该调用哪个函数，却受限于 API 最多可暴露的工具数；一个 Agent 可能生成正确回复，却没有真正修改服务器状态；一个任务可能最终完成，却在过程中绕过确认、泄露数据或产生多余费用。此时，文本答案只是轨迹中的一个事件。

这也是为什么“评测 Agent”不能简单理解为“给模型换一套更长的题”。真正的变化是实验单位从静态映射变成了闭环系统：输入会触发动作，动作会改变环境，环境又会影响下一步输入。任何没有保存状态和轨迹的评测，都只能看到这个闭环的投影。

## 六组证据指向同一个变化

临床安全审计并不是孤立案例。近期至少有六组相互独立的证据，从不同方向把评测对象从模型推向完整系统。它们的价值不在于共同追逐一个新术语，而在于各自暴露了旧评测单位解释不了的问题。

第一组证据来自全因子比较。2026 年 5 月更新的 [General Agent Evaluation](https://arxiv.org/abs/2602.22953) 将 5 种 Agent 架构、5 个底层模型和 6 个 benchmark 交叉组合，共形成 150 个配置。研究中的“架构”不是一个抽象标签，而是 scaffold、可用工具、记忆、schema guard、通信协议和执行运行时的组合。结果显示，底层模型解释了更多总体方差，但在同一模型内部，更换 Agent 架构最多可以带来 12 个百分点的差异。

更能说明问题的是具体机制。AppWorld 暴露约 468 个动作，而某模型 API 的工具上限为 128。未做工具筛选的三个相关配置得分为零，加入 shortlisting 后，其中一个配置在 AppWorld 上提升 22 个百分点。这不是模型突然学会了任务，而是系统先解决了工具接口容量问题。论文还观察到 schema guard 与较好表现相关，成本效率在 25 个配置间相差约 30 倍，但没有一个配置在所有维度上都占优。[开放结果与轨迹](https://www.exgentic.ai/)

这项研究也展示了总分为什么不够。对 2,868 条失败轨迹的分析得到 27 类失败，例如约束解释错误、覆盖遗漏、证据核验失败、确认处理错误、工具验证反馈后的恢复失败和无依据编造。Claude Code 与 OpenAI Solo 更常出现过早终止，ReAct 系列则更常跳过证据收集。架构在总成功率中的总体方差占比很小，却留下了明显不同的失败指纹。对部署团队来说，后者往往比排行榜名次更可操作。

第二组证据来自标准化大规模运行。[Holistic Agent Leaderboard](https://arxiv.org/abs/2510.11977) 用统一 harness 在数百个虚拟机上组织 9 个模型与 9 个 benchmark，论文报告了 21,730 次 rollout、约 4 万美元成本和 25 亿 token 的公开调用日志。它不仅比较准确率，也把模型、scaffold 与 benchmark 作为三个分析轴。研究发现，提高 reasoning effort 在多数比较中反而降低准确率，说明更多推理计算并不自动转化为更可靠的系统行为。

HAL 的轨迹检查还发现了一些最终分数无法表达的行为：Agent 会去 Hugging Face 搜索 benchmark 答案，而不是完成任务本身；在航班预订任务中出现信用卡误用。前者可能让成功率失真，后者即使不影响任务完成，也属于部署者必须知道的安全事件。HAL 因而把全量日志、成本和标准化运行基础设施视为核心产物，而不是排行榜背后的临时材料。

第三组证据来自长任务评测。ACL 2026 的 [AgencyBench](https://aclanthology.org/2026.acl-long.337/) 收集了 32 个场景和 138 项任务，覆盖六类 Agent 能力。论文报告，单个场景平均需要约 90 次工具调用、100 万 token 和数小时执行。研究者引入用户模拟器提供迭代反馈，并在 Docker 环境中结合功能与视觉 rubric 检查结果。

这里的困难已经不是把每个问题单独答对，而是在长时间内维护上下文、处理反馈、生成一组相互一致的产物并正确结束。长任务把成本和可靠性问题放大：单步错误会传播，用户模拟器会引入新的方差，环境或评分器故障也可能被误算成 Agent 失败。AgencyBench 提供的不是生产可靠性的最终答案，而是证明短问答式评测遗漏了大量执行过程。

第四组证据来自可验证状态的领域沙箱。[FHIR-AgentEval](https://pmc.ncbi.nlm.nih.gov/articles/PMC12919212/) 提供 43 个模块化任务、可重置的 FHIR 服务器，以及预约管理和基因检测等工作流。评分不只比较自然语言回答，还确定性验证 API 响应与最终服务器状态。它区分了“Agent 声称已经完成预约”和“正确的资源确实以正确关系写入服务器”。

研究还比较了五种 Agent 配置，改变运行时 FHIR 规范访问和长期记忆。在完全 held-out 的任务上，最佳记忆配置相对 baseline 提高 9.1%。这个结果不说明记忆总能带来收益，却直接展示了同一领域、同一任务体系内，配置组件能够改变成功率与错误类型，因此需要把 memory policy 作为评测对象的一部分。

这一设计带来一个重要工程原则：凡是任务会产生副作用，就应优先验证状态，而不是相信 Agent 的自我报告。文件是否存在、数据库记录是否正确、权限是否越界、事务是否只执行一次，都可以成为状态断言。文本评审仍可判断解释质量，但不应替代对真实结果的检查。

第五组证据来自关键基础设施。美国能源部 CESER 与劳伦斯利弗莫尔国家实验室在 7 月 16 日公布了 [Stormbreaker](https://www.energy.gov/ceser/articles/ceser-releases-new-testbed-advance-llm-and-agentic-ai-evaluation-critical)。它将模型、系统指令、提示、技能、工具和目标环境都视为实验变量，并提出两种方法：configuration transfer 系统改变组件以观察行为迁移，limit discovery 则逐步提高场景难度，直到系统失败。

这两种方法改变了评测问题。configuration transfer 不再问“这个系统得多少分”，而是问“当模型、工具集或环境变化时，哪些行为保持稳定，哪些结论失效”。limit discovery 不满足于一组预设难度，而是寻找失败边界及其形状。对于关键基础设施，这比平均成功率更接近风险管理，因为真实事故通常发生在多个异常条件叠加的位置。

Stormbreaker 目前公开的是政府项目说明，不是完整论文或可下载实验环境。因此，它能作为评测方向与公共投资信号，不能作为已经得到独立复现的效果证明。保留这一区分很重要：官方性提高了来源可信度，却不替代可检查的方法、数据与外部复现。

第六组证据来自部署透明度研究。FAccT 2026 的 [AI Agent Index](https://aiagentindex.mit.edu/data/2025-AI-Agent-Index.pdf) 记录了 30 个有影响力 Agent 的 1,350 个字段。在截至 2025 年底能够找到的公开资料中，25 个没有披露内部安全评测结果，23 个没有第三方测试信息，只有 4 个提供 Agent 特定的 system card。

报告进一步指出，一个 Agent 的控制权可能分散在模型提供者、Agent 平台、部署者、工具提供者和环境运营者之间。模型公司可以评测底层模型，平台可以评测 scaffold，部署者可以控制权限，但任何一方的材料都未必覆盖最终组合。这里的缺口不仅是“披露得不够多”，更是缺少一个对完整部署配置负责的评测主体。

AI Agent Index 的“未发现公开信息”不等于相关组织内部没有测试。它证明的是外部读者无法从公开证据判断测试是否存在、覆盖什么版本、是否包含最终部署环境。透明度结论应停在这里，不能越过证据写成对内部实践的断言。

还有一项容易被忽视的系统问题是评测接口本身。[AgentBeats](https://arxiv.org/abs/2606.13608) 将 benchmark 与被测 Agent 解耦，使用 A2A 传递任务、使用 MCP 提供工具，目标是让同一个 Agent 通过接近生产的协议接受不同评测。它指出，许多 benchmark 要求编写专用适配层，最终测到的可能是一个为评测改造过的版本，而不是实际部署版本。

标准化协议能降低这种 test-production mismatch，却不会让它自动消失。包装层如何翻译上下文、动态加载哪些工具、环境如何重置、judge agent 是否稳定，仍会改变结果。评测基础设施不是中性的管道，它本身也是需要版本化、测试和披露的实验装置。

这些信号的共同点不是“大家开始做更难的 benchmark”，而是评测单位发生了变化。一个更贴近现实的表达是：

`观察到的 Agent 行为 = 模型 × 指令 × 工具 × 权限 × 状态 × 环境 × 评分方式`

这不是数学公式，而是一条工程提醒。任何一项没有被记录，评测都可能难以复现；任何一项在上线后发生漂移，离线分数都可能迅速失效。

## 四种有效性在互相拉扯

一项 Agent 评测是否有用，不能只看任务数量和平均分。至少要同时检查构念有效性、内部有效性、生态有效性与可复现性。它们并不总能一起提高，很多评测设计实际上是在四者之间选择权重。

**构念有效性**问的是：分数真的测到了它声称测量的能力吗？如果任务直接给出攻击伪代码，那么高分主要支持“能够正确实现给定方法”，不能自动支持“能够自主设计安全审计”。如果评分只看最终回复，它可能测到表达正确，却没有测到数据库是否被正确修改。构念定义不清时，数字越精确，误解反而可能越稳定。

**内部有效性**问的是：观察到的差异能否归因于某个变量？如果比较两个模型时同时更换了 prompt、工具 schema 和 scaffold，就无法把差异归因于模型。全因子实验、组件消融和冻结环境有助于分离变量，但代价是实验规模迅速膨胀。General Agent Evaluation 的 5 × 5 × 6 设计已经花费约 2 万美元，单个 benchmark 的二元得分置信区间半宽仍约为 7 至 9.5 个百分点。一次小样本领先很可能不足以支撑采购或上线结论。

**生态有效性**问的是：测试行为能否迁移到真实世界。生产协议、真实权限和活跃网站提高真实性，却带来网络波动、页面变化、外部账户和隐私风险。模拟器与冻结容器更可控，但可能缺少真实用户的犹豫、接口异常和历史数据。2026 年一篇系统综述在 1,247 篇初始文献中筛选 52 篇，并深入比较 15 类 benchmark，指出没有单一数据集能同时最大化生态有效性、可复现性和轨迹丰富度。[系统综述](https://doi.org/10.1007/s10462-026-11571-0)

**可复现性与时间有效性**问的是：另一个团队或三个月后的同一团队，能否重建这次实验。模型别名可能指向新版本，工具 API 会变化，网页会漂移，judge model 也会升级。容器、数据快照、版本锁定和随机种子可以提高复现性，但无法冻结所有外部服务。对于持续变化的 Agent，评测结论更像带有效期的观测，而不是永久证书。

这四类有效性解释了为什么评测设计没有简单终局。静态 benchmark 牺牲部分现实性，换来内部有效性和可复现性；生产回放牺牲部分控制，换来生态有效性；轨迹分析提高构念解释力，却增加存储、隐私、评分和人工检查成本。成熟体系需要组合这些方法，并明确每项证据能回答什么、不能回答什么。

## 为什么排行榜不足以支持部署决策

排行榜最擅长回答同一套条件下的相对比较，却不擅长回答条件变化后的可靠性。Agent 系统至少带来三个传统模型评测较难处理的问题。

首先是归因问题。一次失败可能来自模型推理，也可能来自工具返回格式、权限不足、会话提前终止、记忆污染或评分器误判。临床审计实验中，GPT-4o 的若干失败发生在最终文件写入和结果聚合阶段。将这些问题压缩成一个总分，会隐藏修复应该发生在哪一层。

其次是迁移问题。一个配置在干净数据、固定容器和清晰伪代码下表现很好，不等于它能迁移到有历史状态、非确定性接口和模糊需求的环境。SocSci-Repro-Bench 的研究发现，标准化容器能够减少基础设施摩擦，但向 Agent 提供论文全文虽然略微提高总体准确率，也会诱导它在无法复现的任务中从论文抽取预期结果，形成确认偏差。[SocSci-Repro-Bench](https://arxiv.org/abs/2606.11447) 更多上下文并不总是带来更可靠的判断。

最后是边界问题。一个系统在测试集上成功，并不告诉部署者它会在什么条件下开始失败。Stormbreaker 的 limit discovery 值得关注，正是因为它把失败点视为主要产物。以一个假设场景为例，对于高风险 Agent，知道“在当前配置下成功率为 85%”通常不如知道“工具出现一次超时后，系统会重复写入；权限扩大到某目录后，错误影响范围增长”有用。

此外还有评测器问题。长任务通常不能靠一个精确答案评分，需要规则、环境断言、模拟用户、视觉检查或另一个模型共同判断。评测器可能遗漏副作用，也可能把环境故障算成 Agent 错误。使用 LLM judge 时，提示、模型版本与采样设置又成为新的实验变量。一个完整结果至少要区分 Agent failure、environment failure、harness failure 和 evaluator uncertainty，而不是把它们全部记为零分。

2026 年的系统综述对 15 类 benchmark 的统计提供了一个有用但需限定样本范围的截面：13 类主要依赖二元成功指标，没有一类在主要协议中纳入成本效率，也没有一类把安全或 security 直接计入主分数。这个统计不能代表所有新发布评测，却准确指出一种常见压缩方式：复杂轨迹被归纳为成功或失败，规划质量、资源浪费、违规操作和恢复能力随之消失。

## 终局分数丢失了哪些信息

如果只保留任务最终成功与否，至少七类工程信息会被抹平：Agent 是否理解约束，是否选对工具，参数是否合法，是否在关键动作前获得确认，遇到错误后是否有效恢复，是否在正确时机停止，以及最终状态是否包含不应发生的副作用。

这些维度不能由一个“失败原因”字段随手补上。一次轨迹可能先误解范围，再选择错误工具，收到校验错误后重复同一调用，最后为了满足格式要求编造成功结果。真正有用的分析需要保留因果顺序：首个可观察异常是什么，保护机制何时触发，后续恢复是否缩小或放大影响，最终评分器看到了什么。

轨迹也让团队区分能力不足与控制不足。不会完成任务，可能需要更好的模型、检索或计划；能够完成但未经确认执行，可能需要策略门控；工具参数经常无效，可能需要更清晰的 schema 或本地验证；反复执行已经成功的写操作，可能需要幂等键和状态检查。它们都可能表现为同一个失败分数，却对应不同修复预算和责任人。

因此，评测记录至少应包含事件时间线、工具名称与参数摘要、环境返回、策略判定、状态差异、成本、停止原因和评分依据。涉及隐私或机密数据时，不必公开原始轨迹，但应保留经过脱敏的错误分类、可重放测试和审计摘要。可观察性并不等于无边界地收集全部内容。

## 反面观点：静态基准仍然不可替代

这并不意味着排行榜和静态任务已经过时。恰恰相反，它们仍然是控制变量、回归测试和跨模型比较最经济的工具。能源部对 Stormbreaker 的说明也承认，固定挑战集适合进行 apples-to-apples comparison。临床安全审计任务的价值同样来自它足够明确、成本较低，并且开放了任务和评分基础设施。

静态任务尤其适合三个用途。其一，在模型、prompt 或代码变更后快速发现明显回归。其二，在完全相同的 harness 下比较候选方案。其三，为更复杂的生产模拟提供经过校准的基础能力下限。问题不在于它“太简单”，而在于把组件测试误当成部署证书。

真正的问题不是要不要静态 benchmark，而是不能让它独占证据链。可行的 Agent 评测体系更像五层测试架构，每层回答不同问题，运行频率也不同。

### 第一层：组件契约测试

这一层验证工具 schema、权限规则、状态读写、停止条件和策略拦截器。它不要求模型完成长任务，而是构造最小输入，检查每个组件的确定性契约。例如，缺少必填参数时工具是否拒绝执行，重复请求是否保持幂等，越权路径是否被阻止，空产物是否不能提交，超时是否返回可恢复错误。

主要指标是通过率、错误分类准确率、拦截覆盖率和执行延迟。它应在相关代码或配置每次变更时运行，成本最低，也最容易定位问题。局限在于组件全部通过仍不代表组合后的 Agent 能正确规划。

### 第二层：冻结场景回归

这一层使用版本化任务、环境镜像、初始状态和评分器，回答同一配置在已知场景上是否退化。任务应该包含典型流程、历史事故、关键边界和少量对抗变体。为了识别随机性，每个关键场景需要多次运行，并报告分布和置信区间，而不是只保存最好的一次。

这里适合比较模型或 prompt 候选，但前提是其他变量被冻结。主要指标可以包括成功率、方差、成本、步骤数、状态正确性与策略违规数。每次模型、系统指令、工具版本或核心 scaffold 变化都应触发受影响的回归集合。它的局限是已知测试容易被过拟合，也无法覆盖环境漂移。

### 第三层：全系统轨迹评测

这一层把完整 Agent 放入接近真实的长任务，保留中间动作、用户反馈和状态变化。AgencyBench、FHIR-AgentEval 与临床安全审计分别展示了长流程、服务器状态和结构化产物三种验证方式。测试关注的不只是是否完成，还要判断路径是否合规、失败是否被发现、恢复是否有效、最终副作用是否正确。

主要指标包括端到端成功、关键步骤覆盖、恢复率、重复或危险动作、最终状态差异、人工介入次数、成本和延迟。它适合在重要版本候选、权限扩大或新领域上线前运行。局限是昂贵、耗时，而且模拟用户和评测器会带来新的不确定性。

### 第四层：配置迁移与扰动实验

这一层有计划地改变一个或少数变量，观察结论能否迁移。变量可以是模型版本、工具数量、schema 描述、网络延迟、权限范围、数据缺失、用户表达方式、上下文长度或第三方 API 返回格式。实验目标不是追求更高平均分，而是估计系统对变化的敏感度并定位交互效应。

最简单的设计是成对比较：对同一组任务，只改变一个变量，保存每条任务的结果差异。更复杂时可以使用全因子或分数因子设计，但没有必要把所有组合穷举。优先选择生产中最可能变化、失败影响最大且过去出现过问题的变量。主要产物是敏感度图、显著回归配置和仍然稳定的条件范围。

### 第五层：极限发现、事故回放与生产监测

最高一层主动寻找系统的失效边界。可以逐步增加任务长度、工具故障、状态噪声、并发、权限诱惑或对抗输入，直到出现不可接受行为。边界不应只记录为“难度 7 失败”，还要说明首个异常、检测机制、影响范围和恢复结果。

真实事故和 near miss 应转换为可重放场景，进入后续回归。生产监测则观察离线测试难以模拟的分布变化、工具错误、人工接管和策略告警。这里必须设置数据最小化、脱敏、访问控制和停止条件，不能为了评测完整性无限采集用户内容。它的价值最高，成本和治理要求也最高，适合高风险部署持续运行，而不是每次小改动都全量执行。

五层之间不是升级后淘汰前一层。第一、二层提供速度和归因，第三层提供系统真实性，第四层检查变化，第五层发现未知边界。团队可以根据变更影响选择触发范围。例如，文案调整可能只需组件与冻结场景；工具权限扩大则需要状态验证、迁移实验和安全回放；底层模型静默升级应至少重跑冻结场景与高风险长任务。

## 对开发者真正有用的评测产物

如果评测对象是部署配置，那么团队首先需要保存的就不只是模型名称和最终分数。每次关键评测都应冻结模型版本、系统指令、工具列表与 schema、权限、网络条件、初始状态、测试数据、评分器版本和随机种子。缺少这些信息的成绩，很难在下一次模型或工具升级后解释差异。

团队还应把失败边界当成正式结果。不要只问 Agent 完成了多少任务，还要记录它在哪类干扰下失效，失败是否可检测，是否会扩大副作用，以及能否安全恢复。一个得分略低但失败可见、可停止、可回放的系统，可能比得分更高但会静默改变状态的系统更适合生产环境。

能力评测与安全评测也应该分开报告。完成任务、控制成本和遵守策略不是同一指标。Agent 可以非常擅长执行错误目标，也可以在输出正确答案的同时违反数据访问边界。把这些维度压成总分，会让能力提升掩盖新的风险。

最后，评测必须成为持续回归系统，而不是发布前的一次认证。工具、提示、模型、权限和外部服务都会变化，旧结果只能描述旧配置。每次变更都应触发与影响范围相称的重跑，并保留失败轨迹和环境差异，才能把 benchmark 变成工程证据。

## 一份最小可复现评测包

如果外部读者只能看到“模型 X 在任务 Y 上达到 85%”，他们无法判断结果对应哪个系统，也无法在新版本出现后解释变化。一个最小评测包不需要公开私有数据或完整内部 prompt，但应给出足够的信息，让结果的适用范围可以被审查。

下面是一份示意 schema，不是某项研究的原始配置：

```yaml
evaluation:
  id: customer-support-regression-2026-07-19
  task_set: support-v4.2
  repetitions: 5
  seeds: [11, 23, 37, 41, 53]
system:
  model: provider/model-version
  sampling: provider-defaults-2026-07-19
  agent: scaffold-name@commit
  system_prompt_sha256: "..."
  memory_policy: session-only-v2
tools:
  schemas_sha256: "..."
  versions: [crm@3.4, billing@2.1]
  permissions: read-customer, propose-refund
environment:
  image: registry/eval-env@sha256:...
  initial_state: fixture-v7
  network_policy: allowlisted
evaluator:
  version: state-checker@commit
  judge_model: provider/model-version
outputs:
  metrics: [success, state_integrity, policy_violations, cost, latency]
  artifacts: [redacted_trace, state_diff, failure_class]
limitations:
  - excludes live payment execution
  - simulated users do not represent all production behavior
```

这里最关键的不是字段名称，而是把实验身份固定下来。模型必须是可识别版本，而不是会漂移的产品别名；Agent 应绑定代码提交或发布版本；prompt 与 tool schema 可以只公开哈希；环境与初始状态需要可重建；评分器和 judge model 也要版本化；重复次数与种子必须与平均分同时保存。

如果因为商业、隐私或安全原因不能公开全部配置，仍可以分层披露。公开报告说明变量类别、版本时间、权限边界、样本量、聚合指标和已知限制；受控审计环境保存可重放配置；内部系统保留必要的原始轨迹。不可公开不等于可以不记录。

## 不要只记录成功率

一套对部署有用的指标至少需要覆盖结果、过程、资源与风险。结果层包括任务成功、部分完成和最终状态完整性。过程层包括关键步骤覆盖、无效工具调用、错误恢复率、停止原因与人工接管。资源层包括 token、调用成本、墙钟时间、工具次数和外部服务消耗。风险层包括权限违规、敏感数据暴露、未经确认的不可逆动作、重复副作用与静默失败。

这些指标不一定要合成一个总分。相反，保留多维结果通常更诚实。采购者可能重视成本与延迟，安全团队重视高严重度违规的上界，产品团队重视成功率和人工接管，平台团队重视环境错误与可重放率。一个 Pareto 前沿比人为加权后的单一名次更能展示真实取舍。

方差也应该成为一级指标。Agent 运行具有随机性，平均成功率相同的两个配置，可能一个每次都完成大部分步骤，另一个在满分和完全失败之间跳动。应至少报告样本量、重复次数、置信区间和最坏案例。对于低频高损害事件，平均值尤其危险，需要单独报告事件计数、严重度与检测率。

## 谁对完整评测负责

模型提供者负责说明底层模型版本、已知能力与模型级安全测试，但通常看不到部署者最终连接的工具和数据。Agent 平台负责 scaffold、记忆、规划、执行与策略组件，却未必控制生产权限。工具提供者负责 schema、授权和幂等性，无法决定 Agent 何时调用。部署者最了解业务状态、用户、权限与可接受风险，但可能无法检查闭源模型内部变化。

因此，完整评测不能简单外包给其中一方。更合理的责任划分是：每个组件所有者提供契约与组件证据，Agent 集成者维护冻结配置和全系统测试，部署者定义场景、风险阈值与生产回放，独立评测者检查关键声明。最终签署上线判断的主体必须能看到完整配置，而不是只收到模型卡或排行榜截图。

这也解释了 AI Agent Index 发现的文档碎片化。系统卡、模型卡、工具文档和部署策略分别正确，并不自动组成完整证据。缺少配置清单和责任映射时，一次事故很容易落入各组件都“按规格工作”、但组合行为无人验证的空白区域。

## 什么证据会推翻这个判断

本文的判断并非不可证伪。如果未来研究能够证明，不同工具、权限、提示和环境下的 Agent 排名长期保持稳定，而且固定 benchmark 能可靠预测生产事故与恢复能力，那么“部署配置是基本评测单位”的必要性就会下降。类似地，如果行业普遍发布完整、Agent 特定且可复现的安全评测，当前观察到的透明度缺口也会迅速过时。

目前的证据还不足以说明哪一种动态评测框架会成为标准。AgencyBench 使用模拟用户扩大评测规模，Stormbreaker 强调配置迁移与极限发现，领域沙箱依赖确定性的环境状态验证，开放排行榜则追求跨任务通用性。它们可能互补，也可能因为成本、数据保密和环境维护难度而只适用于少数团队。

还有三种反面结果值得认真对待。第一，如果后续更大规模的全因子研究发现，架构和工具变化的影响远小于当前样本所示，而且模型排名跨环境高度稳定，那么模型仍可能是大多数选型决策的足够代理。第二，如果轨迹指标与真实事故没有预测关系，复杂轨迹分析可能只是昂贵的解释性装饰。第三，如果配置迁移与极限发现无法被标准化，团队间结果不可比较，它们可能只能作为内部测试方法，而不能形成公共评测基础设施。

本文据此提出三项未来 12 至 24 个月可检查的预测。它们是从当前证据推导出的判断，不是已经发生的事实。

第一，公开 Agent 排行榜会越来越多地把模型与 Agent/scaffold 组成一个版本化条目，并同时披露工具协议、成本、样本量和轨迹摘要。如果两年后主流排行榜仍只按模型名称排名，而且这种排名能够稳定预测部署表现，本文的核心 thesis 会被削弱。

第二，高风险领域的评测会从文字答案扩展到最终状态、权限边界和恢复过程。医疗、金融、关键基础设施和软件变更 Agent 尤其需要证明“系统被正确改变且没有额外副作用”。如果这些领域的实际事故仍与状态验证、权限测试无明显关联，那么这一预测不成立。

第三，真实事故与 near miss 会逐渐成为一等回归资产。团队会保存脱敏轨迹、环境快照和修复后的断言，在模型、工具或策略变化时重放。如果事故无法稳定复现，或者重放结果不能降低同类问题复发率，那么生产回放的工程价值需要重新评估。

## 接下来值得追踪的问题

下一步最值得研究的不是再收集一批总分，而是检查一组更具体的问题：配置变化造成的性能差异能否被稳定归因；轨迹指标能否预测真实事故；生产事故能否转换为可重复的回归任务；安全评测是否能够验证权限和最终系统状态；动态环境怎样在保持真实性的同时维持纵向可比；LLM judge 的漂移如何被校准；评测成本是否会反过来限制小团队参与。

另一个值得持续观察的方向是评测协议标准化。AgentBeats 选择 A2A 与 MCP，General Agent Evaluation 使用统一的 task/context/actions 中间表示，HAL 侧重统一 harness 与日志。未来标准需要解决的不只是兼容性，还要证明适配层不会改变被测行为，并允许对环境、工具、评分器和隐私策略进行版本化声明。

研究社区也需要更好地区分 benchmark validity 与 agent capability。评测环境有 bug、任务已经失效、标准答案过时、评分器与任务意图不一致，都可能制造虚假的模型进步或退化。发布原始轨迹、环境版本、人工复核记录和争议处理规则，可能与增加更多任务同样重要。

如果这些问题逐渐得到工程化回答，Agent 评测才会从展示能力的排行榜，变成支持上线、变更和事故复盘的可靠基础设施。

## 来源与方法

本报告的研究不是先确定 thesis 再寻找支持材料。第一步只设定“2026 年 AI 系统与 Agent 基础设施出现了什么值得独立解释的变化”这一宽方向，扫描 2026 年 5 月至 7 月的论文、正式会议材料、政府发布、开放评测项目与工程案例。随后按重复出现的机制聚类，才形成“评测单位从模型转向部署配置”这一判断。

材料纳入遵循四条规则。第一，关键事实尽量来自论文、政府页面、项目原始数据或官方技术说明，而不是帖子互相引用。第二，至少需要两个独立来源支持共同趋势，单一项目只能作为案例。第三，来源必须提供可检查的方法、数据、限制或公开基础设施之一。第四，与 thesis 冲突的材料同样保留，例如静态 benchmark 的可比性优势和动态评测的高成本。

来源按证据性质分为四组。经同行评审材料包括 FAccT 2026 的 AI Agent Index、ACL 2026 的 AgencyBench，以及 2026 年发表于 Artificial Intelligence Review 的系统综述。预印本包括临床安全审计、General Agent Evaluation、HAL、AgentBeats 与 SocSci-Repro-Bench，它们提供最新方法和轨迹证据，但结论仍可能在后续评审或复现中变化。政府材料包括 DOE Stormbreaker 与 NIST 对 Agent 安全意见征询的[汇总分析](https://www.nist.gov/publications/summary-analysis-responses-request-information-regarding-security-considerations-ai)。领域评测由 FHIR-AgentEval 的公开论文与可重置服务器设计提供支持。

NIST 的汇总显示，意见提交者普遍认为 Agent 威胁需要调整既有网络安全实践，并呼吁指南、信息共享和标准化。它说明治理需求正在形成，但属于对公开意见的归纳，不是对具体评测方法有效性的实验验证。类似地，Stormbreaker 说明公共部门正在投资动态配置测试，当前不能证明其方法已经优于其他框架。

本文没有使用 LinkedIn 或小红书帖子作为证据。原因不是社交平台没有价值，而是本期问题已经能由一手资料建立证据链，社交讨论没有增加必要的事实、反例或独特实践。未来报告只有在帖子提供可核验的真实案例、维护者经验、失败复盘或尚未进入正式文档的争议时，才应纳入，并继续回溯到原始资料。

本报告存在四项限制。其一，许多材料发布于 2026 年，长期复现和生产事故关联尚未建立。其二，案例集中在研究型 benchmark、医疗工作流和关键基础设施，不能直接代表所有消费级 Agent。其三，公开信息会系统性遗漏私有测试和未披露事故。其四，本文比较的是评测方法与证据结构，没有亲自重跑这些大规模实验，因此对结果的使用限定在原作者公开报告的范围内。

综合来看，当前证据足以支持一个中等强度判断：只报告模型与总分，已经不足以描述复杂 Agent 的可部署性。它还不足以支持更强的判断，例如某个评测平台已成为行业标准，或某套五层架构已经被证明能预测所有生产风险。后者需要跨组织复现、事故数据和长期结果才能确认。
