为什么 eBPF verifier 报错难修：丢失的证明、模糊的消息与 bpfix

bpfix 研究 235 个可复现的 eBPF verifier 拒绝案例，解释终端报错为何藏住真正修复点，以及证明感知诊断如何指导修复。

eBPF verifier 报错看起来精确到一条指令，开发者却常常改了那一行还是过不了验证。内核报告的是验证停止的位置，真正的修复点往往在更早的代码里，程序在那里丢掉了后续访问所需的指针类型、标量范围、生命周期或来源证明。

bpfix 论文 直接研究了这个落差。在 235 个可复现的 verifier 拒绝案例中，47% 的 errno 是 EINVAL，一个归一化后的终端错误字符串最多对应 9 种不同根因。verifier log 里虽然有有用状态，但终端报错只是一段证明过程的最后一帧。本文介绍 bpfix 研究和工具，包括当前实现尚未解决的案例。

eBPF verifier 报错本质上是证明失败

eBPF 程序进入内核运行之前，verifier 必须证明每一条执行路径都是安全的。它在每条指令处跟踪寄存器和内存的抽象值，逐步建立起包指针是否仍在边界内、map 值指针是否来自正确辅助函数、dynptr 是否仍然有效、标量范围是否足够支撑后续访问等事实。这些事实就是后续指令可以依赖的证明，只有 verifier 能在抽象状态里看到它们时，它们才继续生效。站内的 eBPF 安全概览 更系统地介绍了 verifier 在安全边界中的角色；这篇文章聚焦诊断问题。

这个模型给了 eBPF 安全边界，也改变了拒绝信息的含义。一行 C 代码被拒绝时，源代码错误不一定就在这一行；被拒绝的指令只是 verifier 第一次需要某个证明、但手里已经没有这个证明的位置。

bpfix 论文里的 packet 解析例子很典型。程序先计算 UDP header 指针，把它和 data_end 比较，再读取 dest 字段。

论文图 1 把三种视角放在一起：源代码只是在读取 UDP header，原始 verifier log 停在 R5 invalid mem access 'scalar'，而 bpfix 把终端报错改写成源代码区间和所需证明，即被拒绝的解引用处需要一个 verifier 能识别的包指针。

这段代码看起来已经做了边界检查，但字节码在读取位置已经不再保留包指针证明。终端报错 R5 invalid mem access 'scalar' 说明 verifier 看到的是一个标量，而它需要的是包指针。它没有说明包指针是什么时候变成标量的，也没有区分源代码少了边界检查、编译器 lowering 抹掉了指针来源，还是开发者应该重新派生一个 verifier 能识别的指针。

这里说的证明指 verifier 在抽象状态里能看见的事实。数据包读取需要寄存器仍然被分类为包指针，且访问范围仍在 data_end 以内。map 值写入需要指针来自正确 helper，并且已经做过空指针检查。标量偏移需要范围被收紧到当前对象可以接受的大小。源代码表达的是开发者意图，verifier 接受的是这些事实在抽象状态里是否还存在。

对人来说，这个差别很关键：被拒绝的读取所在行在源代码层面可能完全合理，真正的修复必须在它之前恢复 verifier 能看见的证明。

为什么终端报错太粗

bpfix 从一个 eBPF 开发者很熟悉的问题出发：当 verifier 拒绝程序时，终端报错到底能把修复范围缩小到什么程度？

作者从 Stack Overflow 问题、GitHub issue、GitHub 修复提交和 Linux kernel selftest 中收集了 936 个候选报告，并在 Linux 6.15.11、clang 18、verifier log level 2 这套固定工具链下重新构建和加载它们。log level 2 会让 verifier 在每条指令后打印抽象状态，研究能看到走向终端报错的状态变化。最终能稳定复现为 verifier 拒绝的 235 个候选构成了 bpfix-empirical 数据集。

这个复现过滤很重要。论文纳入的是能在同一套内核和编译器配置下重新构建、加载并被拒绝的案例，所以终端报错、根因标签和修复层次都对应同一个 verifier 行为。

数据的第一层划分已经改变了读日志的方式。235 个拒绝里有 191 个需要修改程序源代码；剩下 44 个拒绝的是正确源代码，修复位置在源代码之外：18 个属于编译器，14 个属于环境，12 个属于 verifier。verifier 层修复意味着源程序在内核侧提升分析精度或调整 verifier 行为后被判定为正确。约五分之一的可复现拒绝没有要求开发者改 eBPF 程序。

191 个源代码 bug 落入 12 类根因；其中 10 类是 eBPF 特有问题。常见错误集中在 verifier 能观察到的证明类别上。证明类别是 verifier 试图建立的事实类型，包括标量范围、dynptr 生命周期、每条路径的包边界、map 查找后的空指针检查和指针来源。不同证明类别需要不同的源代码修复方式，判断拒绝属于哪个类别能把修复目标收窄到终端错误字符串本身无法表达的程度。

根因、证明类别和修复层次回答三个不同问题。根因说明开发者或工具链哪里犯错；证明类别说明 verifier 看见的哪个事实消失了；修复层次说明该改源代码、编译设置、环境，还是 verifier 行为。把这三层分开，终端报错才不会把多种修复压成一个模糊诊断。

这组数据把读日志时最容易混在一起的三个问题拆开。

调试问题论文测量了什么为什么会影响修复
源代码是否需要修改235 个拒绝里有 191 个在源代码层修复，44 个在编译器、环境或 verifier 层修复。终端报错可以描述真实的 verifier 失败，但源代码不一定是该改的层。
缺失的是哪类证明191 个源代码 bug 覆盖 12 类根因，包括标量范围、包边界、指针来源、dynptr 生命周期和 map 查找检查。修复要恢复具体证明类别；单纯让一个错误字符串消失还不够。
终端报错到底有多模糊R# invalid mem access 'scalar' 这个模板覆盖 28 个案例，横跨 9 种根因。同一个终端消息可能需要不同的源代码修改、构建调整或 verifier 侧修复。

终端报错没有携带这种结构。EINVAL 覆盖了全部复现拒绝的 47%，把寄存器和 offset 归一化后，167 个不同错误字符串被折叠成 82 个模板；其中 15 个模板各自跨越多个根因，最常见的 R# invalid mem access 'scalar' 覆盖 28 个案例，横跨 9 种根因。

bpfix 要补的正是这段缺失信息。终端报错仍然是有用的症状锚点，但真正能指导修复的内容藏在走向这个症状的证明生命周期里。

bpfix 重建证明丢失的位置

bpfix 把 verifier log 当成证明轨迹来看。在 log_level=2 下，verifier 会在每条指令之后打印抽象状态，包括寄存器类型、标量范围、指针来源、引用计数等分析过程中的信息。终端报错标出拒绝点，但前面的状态往往记录了相关证明什么时候出现、保存了多久、又在哪里变得不再兼容后续操作。

bpfix 先把终端报错和逐指令抽象状态解析成归一化的证据流，把寄存器编号和 offset 抽象掉，使同一类证明在不同程序里都能被识别。接着从证据流里识别缺失的证明类别，包括指针来源、包边界、标量范围或 map 值派生，并在日志里追踪这个证明。最终诊断会报告所需证明、相关源代码区间、能观察到的丢失点，以及重新建立证明的提示。

这个工作流的重点在于 bpfix 不需要先完整理解开发者在源代码层的全部意图。证据来自 verifier 自己打印的抽象状态，不依赖对开发者意图的猜测，所以它可以处理此前没见过的程序。有源代码和对象元数据时，bpfix 还能把结果映射回开发者能读的源代码区间。

诊断对象看起来相近，但调试意义不同。

诊断对象它回答的问题为什么重要
被拒绝的操作verifier 在哪条指令停下症状，读日志的锚点。
所需证明这条操作当时需要哪个事实说明修复必须恢复哪个 verifier 义务。
丢失点这个事实在哪里消失往往比终端报错更接近真正的源代码或 lowering 问题。
修复层次该改源代码、编译设置、环境，还是 verifier 行为避免在 C 代码已经正确时继续改 C。

论文里的 map 值案例展示了另一类证明。被拒绝的程序把 map 对象本身当成普通内存，把 &globals 转成值指针再写入。终端报错指向写操作，但缺失的证明出现在更早的指针派生处。

程序可以拿到 map 对象指针，却仍然没有写 map 值所需的证明，因为 verifier 期待的是通过 helper 派生出来的值指针。修复动作跟着这个所需证明走。

这个例子有用，是因为终端报错只停在被拒绝的访问上，而真正的修复是一小段建立证明的协议：先查找 map 元素，再检查返回指针，最后通过这个指针写入值。

回到包解析例子，bpfix 报告被拒绝的读取需要一个 verifier 能识别的包指针。日志早先把那个寄存器显示成带边界的包指针，后面却在读取之前把同一个值显示成标量。这个转换点才是有用的调试目标。

同一个思路还能拆开终端报错相同、修复层次不同的案例。有的程序从来没有通过辅助函数派生正确的 map 值指针，需要改源代码；而另一个程序的源代码本身是正确的，但编译器 lowering 生成了 verifier 无法继续识别指针的字节码，修复属于编译或构建设置。两者都可能以 invalid mem access 'scalar' 结束，但不该用同一种方式修。

编译器 lowering 案例把修复层次讲得更具体。论文里有正确的源代码表达式，在某个编译设置下会把 context 字段读取降成 verifier 无法识别指针来源的标量。修复是调整编译选项，让字节码保留 verifier 能看见的指针信息；C 源代码本身可以保持不变。

bpfix 的关键价值在于定位。更清晰的排版当然有用，但诊断信息真正改变调试过程的地方是说清楚失败的证明类别，并指出哪个程序转换让这个证明不可用了。

0.1.x CLI 保持职责范围有意窄。它读取开发者现有工作流产生的 verifier、build、bpftool、libbpf、Aya 或 BCC 日志；可选的对象分析能添加控制流上下文。默认路径不执行加载命令、不替换内核 verifier、不检查已通过程序的语义、不自动编辑源代码。bpfix 提供结构化的修复证据；开发者或修复 agent 仍然拥有修改权，并须在内核和程序测试上验证修复。

这对 LLM 修复为什么重要

当模型尝试修复被拒绝的 eBPF 程序时，verifier 的诊断落差会变得更明显。经验丰富的人有时能从日志里推断缺失的 verifier 状态，但 LLM 只能使用输入文本里给它的信息，而原始终端报错留下的可选修复路径太多。

为测量这个影响，论文构造了 bpfix-bench，包含 75 个源代码级 verifier 修复任务。其中 40 个围绕必要 verifier 证明构造，意思是该任务的 verifier 拒绝只能通过在 eBPF 程序里重新建立某个特定证明类别来解决。另外 35 个从 Cilium、xdp-tools 和 bpftime 等开源项目中最小化得到。每个任务都有独立于 bpfix 的可执行测试套件，修复是否成功由 kernel verifier 和测试套件共同判定。一个修复必须通过 kernel verifier、功能检查和源语义检查才算成功。

这让 bpfix-bench 比把编译成功作为单一标准的补丁基准测试更严格。一个候选修复先要返回程序、完成编译、通过内核 verifier 加载，再通过功能测试，最后还要通过源语义检查，避免模型用删除逻辑或改变原程序意图的方式绕开错误。如果模型通过删除被拒绝的代码路径来让 verifier 不报错，功能测试或语义检查会失败，因此这个基准测试测的是诊断能不能帮助模型恢复同一个 verifier 可见证明，并尽量保留原程序应该做的事情。

论文评测了 Qwen3.6 27B、GLM 5.2 和 Qwen2.5 3B，只给原始 verifier log 时，三个模型的一次性修复成功率只有 0 到 37%，把输入换成 bpfix 诊断后，一次性修复率提升了 11 到 21 个百分点。以论文的主模型 Qwen3.6 27B 为例，75 个任务里的一次性成功数从 22 个升到 38 个，允许一次带失败信息的重试后，同一个模型从 30 个成功升到 44 个。

图里每个模型对应一组柱子，左边是原始 verifier log，右边是 bpfix 诊断，所以它传达的是定位能力。输入程序和测试套件没有变化，变化的是诊断文本从原始 verifier log 变成了 bpfix 对缺失证明和相关区间的解释，当成功率随这个干预上升时，证据指向的是更好的修复目标。

失败阶段的拆分说明了收益来自哪里。bpfix 诊断主要减少 verifier 加载失败和源语义检查失败，这两个阶段都要求修复重新建立 verifier 能看见的证明，同时保留原程序的源语义。对 Qwen3.6 27B 和 GLM 5.2 来说，编译失败在两种输入下都很低，所以这部分收益指向 verifier 诊断，普通 C 代码生成在这里没有主导失败。对 Qwen2.5 3B 来说，原始日志没有产生任何一次性成功修复，bpfix 诊断却产生了 8 个成功修复，并消除了三个上下文窗口失败，这些失败发生在完整原始日志超出小模型输入容量的时候。

这个现象说明，在这个基准测试里真正改变修复结果的是输入文本里的证明上下文，而输入程序、模型和测试套件保持不变。

verifier 失败后应该先看什么

bpfix 改变的是第一个调试问题，开发者不该只盯着被拒绝的指令，而应该先问这条指令需要什么证明，例如数据包读取需要包指针来源和有效边界，map 值写入需要通过正确辅助函数派生出来的指针，dynptr 访问需要对象生命周期仍然符合 verifier 的模型。

这个视角也给了原始 verifier log 更好的阅读顺序，先看被拒绝的操作，再看它需要的证明，然后沿着抽象状态往前找，直到看到这个证明出现、消失，或者从来没有建立。走到这一步，修复层次通常就清楚了，因为源代码 bug、编译器 lowering、环境问题和 verifier 限制可能以同一个终端字符串结束，需要的修复却并不一样。

一次实用的日志阅读可以按四步走。

  • 从被拒绝的操作开始，先说清楚它是哪一种访问。
  • 把这种访问翻译成所需证明，例如包指针来源、map 值指针、标量范围或仍然有效的 dynptr。
  • 沿着抽象状态往前找，直到看到证明出现、消失，或者从来没有出现。
  • 根据这个转换判断修复层次，因为源代码少检查、编译器丢掉指针信息、verifier 精度不够，会在日志里留下不同轨迹。

这个阅读顺序也会越过源代码。如果抽象状态显示源代码确实建立过某个证明、而字节码把它丢掉了，下一步该看的就是编译器输出，C 代码反而可能已经表达了正确意图。
