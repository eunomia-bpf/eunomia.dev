# Lobsters Platform Preferences

## Source Notes

- Lobsters is a computing-focused link aggregation and discussion community.
- Its guidelines ask whether a story will improve the reader's next program,
  deepen understanding of the last program, or remain interesting years later.
- Self-promotion should not exploit the site as a write-only traffic channel;
  Lobsters gives a rule of thumb that self-promo should stay below a quarter of
  one's stories and comments.

## Style And Positioning

- Submit only the technically durable subset of our work.
- Keep the 80% contribution / 20% promotion ratio, and be stricter when recent
  account activity is mostly our own links.
- Favor detailed engineering, source code, postmortems, unusual runtime
  behavior, compiler/kernel/security work, and deep implementation writeups.

## Audience

- Computing-focused practitioners who value precision, depth, and useful
  disagreement.
- Less receptive to broad entrepreneurship, company news, generic AI hype, or
  product announcements without technical substance.

## Rewrite Rules

- Keep titles factual and close to the source.
- Choose tags from the platform's predefined list; do not force a story onto
  Lobsters when no tag fits.
- Prefer canonical eunomia.dev, paper, or GitHub URLs without analytics
  parameters.
- Do not add a marketing abstract. Let the source and comment discussion carry
  the value.

## Submission Decision Workflow

- Search Lobsters and the web for duplicate URLs, previous project posts, and
  adjacent discussions before recommending a submission.
- Check tag fit before writing the title. If no tag fits, the story probably
  does not belong there.
- Prefer stories that will remain useful after the launch window: source code,
  implementation notes, measurements, kernel/runtime/security analysis, or a
  postmortem-style lesson.
- Use one canonical URL and no campaign parameters.
- Run the anti-AI pass on comments: remove generic compliments, tidy three-part
  summaries, shallow "interesting perspective" replies, and vague AI hype.

## Short-Form Style

- Lobsters short-form work is title, tags, optional context, and high-signal
  comments.
- Keep the title technical and durable. Avoid launch adjectives, all-caps,
  emojis, and "we built" unless the source title already uses it naturally.
- Use `release` or `show` tags only when the current tag list and story type
  support them. FLOSS announcements still need technical substance.
- If commenting as the author, disclose that relationship and add implementation
  detail, not a pitch.
- Avoid AI-tell phrasing: generic compliments, tidy three-part summaries,
  shallow "interesting perspective" replies, and arguments that ignore the
  strongest reading of the other comment.
- Consider account constraints before submitting: new-user restrictions, unseen
  domains, and tag limitations can affect whether a browser submission works.
- Author comments should disclose affiliation when relevant and add one concrete
  implementation detail, limitation, or reproduction path.
- If a story is mostly a product announcement, route it to another platform or
  rewrite the source into a technical artifact first.

## Browser Checks

- Read the current guidelines and tag list before submitting.
- Search for duplicate URLs or related posts within the recent window.
- Check whether the account has restrictions for new domains or tags.
- Confirm the story is not just a product announcement.

## Post-Publish Follow-Up

- Participate as the author/maintainer when useful, but do not dominate the
  thread.
- Answer technical questions with evidence and implementation details.
- Accept corrections gracefully and update canonical docs when needed.
- Treat private messages as user-controlled; draft replies only when asked.
