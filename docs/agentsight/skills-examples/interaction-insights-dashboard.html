Interaction decision

下一批 agent 任务可以继续交给 agent，但要先加协作契约和验证门槛。

本报告基于 8 个近期非本任务的本地 coding/debugging 会话摘要，观察消息流、工具调用、纠正信号和验证痕迹。 目标是帮助 agent owner 决定：下一次怎样减少返工、提高总结可信度，并更早发现需要人工 checkpoint 的任务。

读者/决策契约：面向 agent owner 和 team lead，判断哪些协作规则、验证要求和任务分流策略应在下一轮 agent run 前更新。
执行判断
可继续，但要加门槛
  • 长会话和高工具调用任务先写清交付判断。
  • 连续纠正后暂停执行，重新确认目标和非目标。
  • 最终总结必须列出已运行、未运行和推断项。
  • 工具失败或重复尝试超过阈值时进入 checkpoint。

下一轮最该改的 5 件事

1

开局写清“这次要做什么判断”

每个任务开始时先确认交付判断：能否合并、能否放行、是否需要复跑、还是只需要定位问题。这样可以减少 agent 把探索、实现和审查混在一起。

证据：8 个会话中有 360 条用户消息和 270 次工具调用，长会话更容易出现目标漂移和后续纠正。

2

把连续纠正设为 checkpoint 信号

当用户连续重设范围、否定方向或要求重做时，agent 应停下来压缩当前理解，而不是继续扩大编辑或重复运行命令。

证据：摘要里出现 74 个纠正/重设类信号，集中在长会话和工具密集会话。

3

把验证结果写成可审查状态

最终总结不要只说“完成”或“已验证”。应列出检查项、状态、范围和缺口，让 reviewer 能直接判断还差什么。

证据：工具结果里有大量错误样式信号；没有统一状态模板时，用户需要重新追问真实验证范围。

4

工具计划从“全量尝试”改成“窄验证优先”

对 build/test/debug 类任务，先运行最小复现和局部检查；只有在局部通过后再扩大范围，避免 Bash/Edit/Read 循环放大。

证据：工具类别以 Bash、Read、Edit 为主，长会话中工具调用密度明显更高。

5

把高风险任务路由到更强采集

当任务涉及文件边界、网络访问、权限变化或客户交付，交互日志只能解释协作过程，不能证明系统副作用。此时应补充系统级证据。

证据：本报告没有读取文件内容、网络 payload 或进程级因果链；系统副作用需要独立证据。

证据与推断

观察项 摘要证据 解释 下一步
长会话 最长样本约 6 小时；另有多个 20-50 分钟级任务。 长会话容易把定位、实现、验证、总结混在一起。 超过时间或工具阈值时重写目标和剩余风险。
工具密集 工具调用 270 次；主要类别为 shell、读取和编辑。 适合设置窄验证优先和重试预算。 第一次失败后记录假设，第二次失败后缩小范围。
纠正信号 纠正/重设类信号 74 个，集中在复杂会话。 用户可能在不断修正 agent 的目标模型。 连续纠正后暂停执行并确认当前理解。
证据缺口 未读取原始全文、文件 diff、网络内容或进程因果链。 本报告能解释协作摩擦，不能证明系统副作用。 高风险运行需要系统级报告配合。
