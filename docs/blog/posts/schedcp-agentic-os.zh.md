# 让 AI Agent 调优 Linux 调度器：SchedCP 如何接入 sched_ext

AI Agent 能写调度器代码，但写出代码和真正调优调度器之间有一段距离。Linux 调度器看到的是可运行任务、唤醒、优先级和 CPU 时间片，而应用真正关心的目标可能是构建耗时、P99 延迟、吞吐量，或者批处理队列里那个最长任务什么时候能结束。成功的调度器调优需要明确优化目标、控制开销、避免饥饿，并在测试环境中保持稳定。

SchedCP 论文的切入点正是这个差别。它把 AI Agent 放进一个受控的调度器控制平面：Agent 负责理解工作负载目标和生成策略，系统接口负责提供观测、复用已有策略、验证生成结果，并通过 Linux `sched_ext` 部署。论文给出的证据仍处于初步阶段，既有成功案例也有失败：Agent 改进了内核编译性能，能从一次错误的调度器选择中恢复，同时把生成成本降低了 13 倍，但评测规模有限，作者也明确表示还需要完整基准测试。

## 调度器调优的问题在于目标

Linux 的 EEVDF 调度器面向多样化的工作负载提供通用策略。这种通用性有价值，却也留下了语义鸿沟。一个并行内核构建、一个有突发流量的服务、一个由许多短任务和一个长任务组成的批处理作业，在内核看来都是一组等待运行的任务，但它们需要的调度取舍并不相同。

过去基于强化学习的调度器通常需要人先定义问题空间：要观测哪些特征、允许调哪些参数、优化目标函数是什么，然后再训练模型。LLM Agent 看起来更像一个性能工程师，它可以读代码、调用工具、查看性能剖析、推断工作负载结构，再生成 eBPF 调度器。但论文里的动机实验说明，直接把终端和 root 权限交给 Agent 远远不够。

作者让 Claude Code 从空目录开始，根据「write a FIFO scheduler in eBPF」生成调度器。三次尝试里只有一次成功。一次在 6 分钟后产出伪代码，另一次在 8 分钟后写成了调度器追踪器。成功的那次用了 33 分钟、221 次 LLM API 调用和 15 次以上迭代，成本约 6 美元。更重要的是，生成出来的代码在一些工作负载上比 EEVDF 更慢，测试时需要 root 权限，可能导致系统崩溃，也没有回退机制。

这组结果说明代码生成只解决了表层问题。真正的调度器调优还需要目标识别、效果观测、安全边界、验证流程和失败后的恢复路径。

## SchedCP 作为控制平面

SchedCP 把 AI 逻辑和系统接口分开。Agent 处理「要优化什么」：理解工作负载、选择策略、生成配置或代码。SchedCP 处理「怎样观测和行动」：提供性能剖析信息、查找已有调度器策略、验证生成结果，并把通过验证的策略部署到 `sched_ext`。

![SchedCP 控制平面设计](imgs/schedcp-design.png)

论文把 SchedCP 实现为 Model Context Protocol（MCP）服务器。这里的 MCP 重点在于接口约束：Agent 通过明确的工具读取允许读取的信息，把想部署的代码交给系统侧验证，而不是为了试一个调度策略就拿到整台机器的无限制 root 权限。

控制平面包含三个核心服务：

- **Workload Analysis Engine** 分层提供工作负载信息。Agent 可以先拿到 CPU、内存等较低成本摘要，再根据需要请求 `perf`、`top`、动态 eBPF 探针等更深入的观测结果，并通过部署后的吞吐量或延迟变化拿到反馈。
- **Scheduler Policy Repository** 保存可执行的 eBPF 调度器程序及其元数据，包括自然语言描述、目标工作负载和历史性能数据。Agent 因此可以先检索已有策略或可组合的代码片段，再决定配置、修补还是生成新策略。
- **Execution Verifier** 在「模型生成了代码」和「系统允许部署」之间加上一道门槛。论文描述的验证流程包括内核 eBPF 验证器、面向调度器逻辑的静态检查，以及在安全微虚拟机中进行的动态验证。验证通过后，系统可以签发部署令牌，并在灰度部署中监控性能，必要时通过熔断机制回退。

这是论文的核心架构主张：Agent 之所以有用，是因为系统给它提供了狭窄、可度量、可逆的行动方式，而不是因为它足够聪明就可以被信任。

## sched-agent 的闭环

SchedCP 是控制平面，`sched-agent` 是论文中构建在这个控制平面上的多 Agent 优化器。它使用 Claude Code 的子 Agent 机制，把调优过程拆成 Observation、Planning、Execution 和 Learning 四个角色。

Observation Agent 先建立工作负载画像。它从进程名、命令行等便宜信息开始，如果线索不足再请求更深的性能剖析。论文中给出的内核编译画像是一个典型例子：CPU 密集型的并行编译，包含大量短生命周期进程和进程间依赖，优化目标是缩短总完成时间。这样的自然语言画像把「系统正在运行什么」转成了「调度器应该优先考虑什么」。

Planning Agent 根据画像选择策略，先尝试配置已有调度器，再考虑补丁或从策略库中组合新的调度器。Execution Agent 负责生成配置或代码，并把结果提交给 Execution Verifier；如果验证失败，它要根据错误继续修正。Learning Agent 读取部署后的性能反馈，把有效配置、部署上下文和反例写回策略库，让下一轮搜索少走弯路。

这个循环有点像强化学习，但论文更强调上下文强化学习：它在当前 Agent 会话和策略库上下文中根据测量结果迭代改进，而不是为每个工作负载重新训练一个模型。

## 初步评测结果

论文的评测规模有限但给出了具体数字。实验使用两台机器：一台是 86 核 Intel Xeon 6787P、758 GB RAM、Linux 6.14，另一台是 8 核 Intel Core Ultra 7 258V、30 GB RAM、Linux 6.13。Agent 使用 Claude Code（Opus 4），每个测试场景测 3 次取平均。作者说明所有实验都生成了可工作的自定义调度器配置或 eBPF 程序，同时也明确表示未来还需要完整基准测试。

在 Linux 内核编译实验中，工作负载是 `tinyconfig` 和 `make -j 172`。SchedCP 第一次选择 `scx_rusty`，相对 EEVDF 获得 1.63x 加速；迭代后选择 `scx_layered`，平均构建时间从 13.57 秒降到 7.60 秒，总体达到 1.79x 加速。图中用于对比的 basic RL scheduler 为 13.79 秒，略低于 EEVDF。

`schbench` 更能体现闭环的意义，因为第一次选择是错的。Agent 初始配置使用 `scx_bpfland`，吞吐量从 EEVDF 的 910 req/s 降到 741 req/s，P99 延迟从 40.3 ms 增加到 46.1 ms。经过 3 轮反馈迭代后，Agent 选择了 `scx_rusty`，吞吐量达到 1452 req/s，P99 延迟降到 19.1 ms。论文将其报告为相对 EEVDF 吞吐量提升 1.60x，P99 延迟改善 2.11x。

在新调度器生成实验中，论文使用 8 个批处理工作负载：文件压缩、视频转码、软件测试、数据分析等。这些工作负载都有长尾结构，即 40 个并行任务中 39 个较短、1 个较长。`sched-agent` 识别出这个模式后生成最长作业优先调度策略，平均延迟降低 20%。论文还报告，Claude Opus 能以每次约 0.15 美元的成本正确分类全部 8 个工作负载，而 Claude Sonnet 失败；在 SchedCP 的支持下，每个工作负载的生成成本降到约 2.5 分钟和 0.45 美元，相比朴素生成基线降低 13 倍。

## 评测覆盖的边界

这些结果支持的是一个较窄的结论：在受控实验里，有工作负载分析、策略复用、验证和反馈的 Agent 闭环，能找到有用的调度器配置，能从一次错误选择中恢复，也能为特定长尾批处理场景生成简单有效的策略。广泛的生产可靠性，包括多种内核版本、多代硬件、混合工作负载、各类故障模式以及长期多租户部署，仍需未来的评测验证。

模型依赖也需要谨慎看待。论文使用的是 Claude Code Opus 4，而且在批处理分类实验里 Opus 和 Sonnet 的表现不同。这反而强化了 SchedCP 的设计动机：模型能力会随版本变化，因此验证、度量、权限边界和回退机制更应该留在模型外部，由系统接口负责。

因此，SchedCP 最值得关注的地方在于它的重构方式。调度器调优可以被处理成一个闭环：先识别工作负载意图，再复用或生成策略，通过验证后部署，最后用真实指标更新下一轮决策。对于系统工程师和 AI 基础设施团队来说，这比单次代码生成更接近可落地的 Agentic OS 形态，是从「这个工作负载跟别的不一样」到「这个调度取舍确实有效」的更安全路径。
