Abstract
The widening gap between processor speed and storage
latency has made data movement a dominant bottleneck
in modern systems. Two lines of storage-layer innovation
attempted to close this gap: persistent memory shortened
the latency hierarchy, while computational storage devices
pushed processing toward the data. Neither has displaced
conventional NVMe SSDs at scale, largely due to programming complexity, ecosystem fragmentation, and thermal/power
cliffs under sustained load. We argue that storage-side compute should be reversible: computation should migrate dynamically between host and device based on runtime conditions. We present WIO, which realizes this principle on CXL
SSDs by decomposing I/O-path logic into migratable storage
actors compiled to WebAssembly. Actors share state through
coherent CXL.mem regions; an agility-aware scheduler migrates them via a zero-copy drain-and-switch protocol when
thermal or power constraints arise. Our evaluation on an
FPGA-based CXL SSD prototype and two production CSDs
shows that WIO turns hard thermal cliffs into elastic tradeoffs, achieving up to 2× throughput improvement and 3.75×
write latency reduction without application modification.

Introduction

The growing disparity between processor speed and storage
latency makes data movement between compute and storage
a dominant performance and energy cost in modern systems [8]. Modern OLAP engines, log-structured key–value
stores, vector databases, and deep learning frameworks all
perform substantial work in the I/O path: compression and

decompression, log formatting, predicate evaluation, and
schema decoding have become routine operations on every
read or write. At the same time, datacenter CPUs operate
under strict power and thermal budgets. Socket-level power
caps and aggressive turbo policies cause frequent changes in
core frequency; under sustained load, storage tasks that previously consumed a modest share of CPU time can suddenly
compete with latency-critical application threads, turning
the I/O stack into a primary bottleneck.
The problem is intensifying. Large language model (LLM)
inference deployments routinely spill model weights and
KV-cache into secondary storage, creating sustained sequential reads interleaved with bursty metadata updates [14, 54].
Disaggregated architectures, which separate compute and
memory into independently scaled pools, further increase
the volume and heterogeneity of data movement [17, 49].
Meanwhile, modern asynchronous I/O frameworks such as
io_uring [6] have lowered software overhead to the point
where hardware and placement decisions—not system-call
costs—dominate end-to-end latency. These trends demand
a storage substrate that can absorb I/O-path computation
near the data when beneficial and return it to the host when
conditions change.
Two lines of storage-layer innovation have attempted to
close this gap. Persistent memory (PMem) shortened the latency hierarchy by placing byte-addressable media between
DRAM and flash, reducing the distance data must travel.
Computational storage devices (CSDs) took the complementary approach: pushing computation toward the data to avoid
moving it at all.
Neither has displaced conventional NVMe SSDs at scale.
PMem required a complex dual-mode programming model;
most deployments defaulted to the simpler Memory Mode,
while App Direct mode demanded months of refactoring [26,

A deeper problem undermines all static offload: it is brittle
under sustained load. Device-side processors operate within
tight thermal envelopes; FPGAs consume 5–20× more power
than ASICs [31], and SSD controllers throttle at 65–77◦ C [42].
Our measurements show that SmartSSD exhibits multi-stage
throttling culminating in shutdown at 100◦ C, while ScaleFlux degrades by 60% under sustained writes (§2.1). Once

CXL SSDs appear to offer a middle ground. They reuse
commodity NAND and NVMe controllers but interpose a
CXL controller that presents a coherent memory region to
the host [34, 46]. The host can map a portion of the device
DRAM into its address space, and device cores can operate
over the same region. Coherency means that neither side has
to manually flush or invalidate caches. However, hardware
capabilities alone do not guarantee usable systems: the operating system must present abstractions that fit existing file
and block interfaces while still exploiting coherent device
memory and on-device compute. Most importantly, the system must adapt to dynamic power and thermal conditions
instead of hard-failing when devices throttle.
We propose a different principle: storage-side compute
should be reversible. Rather than committing computation
to one location, the system should migrate it between host
and device as conditions change. This requires a new storage abstraction that decouples compute placement from the
application interface. Prior substrates made such migration
impractical because transferring execution state required
costly DMA across non-coherent memory [41]. CXL changes
this equation: CXL.mem exposes device memory as a coherent, cache-line-addressable region [12]. Shared state need
not move; only lightweight control state (∼KB) migrates. In
this setting, upload—the reverse of offload—becomes cheap
enough to enter the storage fast path.

The Case for Reversible Compute

This section presents empirical observations on representative computational-storage platforms and derives the design
principles behind WIO.
2.1

The Fallacy of Static Offload

Figure 1 shows thermal behavior under sustained sequential
writes on three platforms. In our measurements, SmartSSD
exhibits multi-stage throttling: the NVMe controller throttles at 70◦ C with 50% throughput loss; the FPGA reduces
frequency at 93◦ C, activates clock gating at 97◦ C, and triggers shutdown at 100◦ C [5]. ScaleFlux throttles at 65◦ C with
60% degradation. The root cause is thermal budget asymmetry: enterprise SSDs consume 10–14 W, but adding FPGA
or embedded compute raises power to 25–70 W within the
same form factor [30]. These are not edge cases—sustained
writes arise in LSM compaction, checkpoint, and backup.
Observation 1: Device-side compute under sustained load
becomes a liability, not an optimization. The system must
support dynamic upload of computation back to the host.
2.2

Lessons from PMem and Legacy CSD

PMem failed not because the media was slow—Optane offered latencies within 2–3× of DRAM [58]—but because the
programming model was too complex. App Direct required
PMDK, DAX filesystem support, and explicit persistence
domain management; most deployments chose the simpler
Memory Mode that sacrificed persistence [55]. Platform lockin and a $559 M inventory write-down confirmed market
rejection [23].

Legacy CSDs failed for analogous reasons. Each vendor
imposed proprietary APIs: Samsung required SDAccel, ScaleFlux exposed fixed-function compression, NGD used embedded ARM [51]. Code written for one CSD could not run
on another. Without cache coherency, data shared between
host and device required explicit synchronization—negating
near-data benefits [33].
Why not perform upload on these devices? Because transferring running state over PCIe DMA is slow and blocks I/O
queues; the lack of coherent shared memory makes dynamic
migration impractical. Both generations failed at transparent
integration (no application changes) and elastic placement
(computation not locked to one location). Early active-disk
proposals from the late 1990s recognized the value of pushing
computation to storage [1, 44], but they assumed homogeneous media and modest firmware complexity; the explosion
of flash translation layers, wear leveling, and multi-queue
interfaces in modern SSDs makes that model impractical
without a portable, sandboxed execution substrate.
2.3

The CXL Opportunity and Its Pitfalls

CXL multiplexes three sub-protocols over a single PCIe 5.0
physical layer [12]. CXL.io provides standard PCIe enumeration and DMA. CXL.cache allows devices to cache host
memory with full coherency. CXL.mem—the most relevant
for storage—lets the host load and store into device-attached
memory as if it were local DRAM, with hardware-managed
coherence that eliminates explicit flushes or invalidations.
Earlier coherent interconnects such as CCIX [11] and GenZ [18] pursued similar goals but lacked the critical mass of
industry support; CXL’s alignment with PCIe ensured rapid
adoption by CPU, memory, and device vendors [53]. CXL 3.0
extends the model to multi-headed and fabric-attached devices, opening a path to rack-scale coherent memory [13], but

However, CXL alone is not sufficient. Naive polling-based

Coherent shared memory is the enabler for reversible compute, not the solution itself.

3

WIO Design

WIO is guided by three principles: (1) application transparency—existing applications continue to use standard POSIX
and block interfaces without any code modification, and all
benefits from CXL SSDs are realized beneath those interfaces;
(2) I/O-path logic is decomposed into fine-grained units that
can be scheduled and migrated independently; (3) all execution, whether on host or device, uses a common sandboxed
runtime to avoid per-device toolchains and simplify portability.
3.1

Architecture Overview

Figure 3 shows the architecture. Host applications issue I/O
through POSIX or io_uring; WIO interposes between
io_uring and the page cache, attaching storage actors
to each request’s processing path. The device domain runs
ARM cores with a separate Linux instance and a WASM
runtime. A central 32 GB PMR serves as coherent shared
memory; I/O queues, DMA buffers, and shared actor state
reside here. An FPGA acceleration layer provides engines
for compression (LZ4), encryption (AES-256), and checksumming (CRC32). NAND arrays provide bulk capacity with
asynchronous background draining from PMR to flash.
3.2

storage actor is a small module that consumes one or more
pages or records, consults and updates shared state, and produces transformed output. Unlike general-purpose actors

in a per-request pipeline, receives data from its predecessor,
and forwards results to its successor. This restricted communication pattern simplifies migration—an actor’s interface is
fully determined by its pipeline position—and enables the
scheduler to reason about per-stage cost without tracking
arbitrary message graphs. Examples include decompressors,
integrity checkers, encryptors, JSON parsers, log formatters, and predicate evaluators. Actors are invoked as part
of per-request dataflow pipelines. For example, a read of
compressed, checksummed log segments may invoke an integrity checker followed by a decompressor and a decoder;
a background compaction operation might invoke actors for
key rewriting and index construction.
Each actor instance has two kinds of state. Control state
captures the current execution context: instruction pointer,
call stack, and local variables. This state is small, actorprivate, and can be serialized. Shared state includes long-lived
structures such as counters, histograms, per-range metadata,
LRU lists, and statistics; WIO allocates these structures in
the CXL-mapped PMR so that both host and device actors
can see the same view. During migration, only control state
is moved between host and device. Shared state remains in
place, and ownership is transferred using a small metadata
protocol that ensures only one side writes a given object at
a time.
3.3

WebAssembly and MVVM Execution

To avoid writing separate binaries for host and device cores,
WIO uses a WebAssembly-based execution substrate. Storage actors are compiled to WASM modules that implement a
restricted ABI providing functions for page access, metadata
lookup, and logging. On the host, actors run either inside
in-kernel WASM sandboxes or within user-space helper processes pinned to dedicated cores. On the device, the same
WASM modules run inside small virtual machines hosted
by a MVVM runtime derived from doubleJIT-style frameworks [39].

The MVVM runtime isolates actor instances from each
other and from the rest of the device kernel. Each VM has
a linear memory region, and access to the PMR and devicelocal DRAM is mediated through a narrow syscall-like interface. This structure allows WIO to run untrusted or buggy
actors while preserving device integrity, and it provides a
convenient hook for resource accounting and per-tenant
throttling. The v86-based translator allows reuse of existing
x86 code when necessary by turning legacy binaries into
WASM before execution.
We considered three alternatives to WebAssembly before
settling on it as the actor substrate. Native plugins (shared
libraries loaded into the kernel or device firmware) offer
the best raw performance but sacrifice portability—each ISA
requires a separate build—and provide no memory isolation without hardware support. eBPF [37] is attractive for
in-kernel storage extensions [62], but its verifier restricts
program complexity (bounded loops, limited stack depth, no
heap allocation), making it unsuitable for multi-stage actor
pipelines that maintain variable-sized state across invocations. Lightweight VMs [2] provide strong isolation but incur
millisecond-scale startup and require a guest kernel, which
is impractical for per-request actor instantiation.
WebAssembly [21] occupies a practical middle ground:
it gives WIO one actor format across x86 host cores and
ARM device cores, provides memory-safe sandboxing without kernel privilege via its linear memory model, and keeps
control state compact and easy to checkpoint during drainand-switch. Prior work has shown that WASM execution
overhead is modest for I/O-dominated workloads and primarily affects compute-dense kernels [27]; production runtimes
such as Wasmtime [10] demonstrate that ahead-of-time compilation can close much of the remaining gap. Section 5.6
shows that these trade-offs are practical in our setting: the
cost is concentrated in dense numeric kernels, while the
control, metadata, and data-movement stages targeted by
WIO remain close enough to native for fast-path use. Deviceside actors use ahead-of-time (AOT) compilation to avoid

Live Migration and Asynchronous Durability

When the scheduler triggers migration, WIO uses a drainand-switch protocol to avoid I/O stalls. The protocol proceeds in four steps:
1. New incoming requests are immediately routed to the
destination.
2. The source drains its in-flight requests to completion.
3. Control state is checkpointed into the PMR.
4. A doorbell interrupt notifies the destination, which reconstructs the actor in a fresh WASM sandbox, reattaches
shared state from the PMR, and resumes.
Because shared state resides in coherent memory, no data
copying occurs. No requests are dropped or replayed; the
transition is transparent to applications. Typical control state
is ∼8 KB; the entire migration completes in under 50 𝜇s.
3.5

HOST

Scheduling and Thermal Management

The scheduler samples host metrics (core frequency, RAPL
power, io_uring queue depth) and device metrics (temperature, utilization) every 10 ms. A placement cost function
drives decisions:
• Device temperature > 𝑇high (75◦ C) and host has headroom: upload actors to host.
• Host CPU > 𝑈 high and device is cool: offload actors to
device.
• Both near limits: degrade rate or shed load rather than
migrate.
The scheduler classifies each actor flow as either latencysensitive or best-effort. Latency-sensitive stages (e.g., WAL
writes, metadata lookups) remain on the host unless it is
throttling; background stages (compression, compaction, log
reformatting) are offload candidates. To avoid thrashing, the
scheduler enforces a minimum residency interval: once an
actor is placed, it remains at that location for at least 100 ms
before being reconsidered. Migration frequency is bounded
to at most one actor move per 10 ms scheduling epoch.
This design answers a key question: “isn’t uploading just
moving work back when the device is hot?” WIO provides
a transparent, schedulable OS/storage abstraction: placement
is decoupled from the application interface, migration is
zero-copy and sub-millisecond, and the scheduler considers multiple dimensions beyond temperature alone. The key
difference from ad-hoc thermal management is that WIO
migrates actors—self-contained execution units with welldefined state boundaries—rather than opaque threads or callbacks, and that shared state in PMR does not move during
migration.

either scenario; only the lightweight control state (∼8 KB)
may need to be re-checkpointed.

4

Implementation

WIO runs on a CXL SSD prototype combining an Intel Agilex
FPGA [22] with Samsung enterprise SSDs, integrating 32 GB
PMR and controller DRAM exposed through CXL 2.0.
4.1

CXL SSD RTL Implementation

Figure 4 shows the RTL organization. Two PCIe ×8 cores terminate CXL/PCIe links and expose two BAR regions: BAR0
maps up to 128 TB of CXL.mem-visible VMEM used as the
byte-addressable aperture, while BAR2 provides an 8 GB region holding configuration registers, SSD initialization state
(queue bases, depths, tail/head pointers), and doorbells. A
MEM2NVME bridge translates between CXL.mem load/store
semantics and NVMe queue operations on the underlying
SSD. Internally, the bridge uses ordered Avalon semantics
toward the SSD controller and out-of-order AXI MCDMA
within the accelerator fabric, using distinct read IDs to allow

aggressive reordering while committing responses in-order
at the PCIe core boundary.
The read path for a host access to VMEM proceeds as
follows. The host issues a PCIe read TLP targeting VMEM
(BAR0). The PCIe core receives the TLP, forwards the request over BAM to MEM2NVME, and the MEM2NVME logic
interprets it as a metadata or data access, possibly triggering a write to the SSD’s submission queue tail register in
BAR2. The SSD controller fetches the corresponding SQ entry via its own PCIe/NVMe path, reads or writes flash as
needed, and eventually returns data. The completion travels
back the same path: SSD to PCIe core, then through BAM
to MEM2NVME, which writes the data into VMEM (BAR0)
and generates the completion TLP to the host. From the
host’s perspective, this entire sequence appears as a coherent CXL.mem load, while internally it is implemented as a
sequence of PCIe transactions and NVMe queue operations.
4.2

Host-Side Integration

Within the kernel, WIO modifies the io_uring fast path [6]
to attach per-request metadata that describes the actor pipeline
and references into shared state. The io_uring interface
was chosen because its submission–completion ring structure maps naturally onto the PMR-resident queues: both use
shared-memory rings with producer and consumer pointers,
and both support batched, zero-copy semantics. Each submission queue entry carries a compact descriptor (32 bytes) that
identifies the storage actors to run, their input/output buffers
in PMR, and a handle to a per-request state blob shared between host and device. The descriptor format is extensible:
a 4-bit opcode field selects among predefined actor pipelines
(compress, encrypt, checksum, passthrough), while a flags
word enables optional stages such as integrity verification
or format conversion.
WIO adds hooks in the page cache to allocate certain pages
from the PMR DAX node and to maintain ownership information used during migration: each cache page is tagged with
an actor identifier, placement hints, and an epoch counter
that allows the runtime to detect concurrent relocation. The
epoch counter is critical for correctness under concurrent
migration: if a host thread reads a page whose epoch has
advanced, it knows a relocation is in progress and retries
after the migration completes.
A small in-kernel control plane monitors CPU and device metrics and applies the scheduler’s policies. It samples
per-core frequency, C-state residency, and memory bandwidth through performance counters, and reads temperature
and throughput statistics from the device via CXL.io configuration registers and a lightweight telemetry protocol.
A user-space daemon performs configuration, tracing, and
offline profiling; it can replay recorded traces to evaluate
alternative scheduling policies without disturbing running
workloads.

4.3

CXL-Aware MONITOR/MWAIT Path

WIO’s host runtime exploits x86 MONITOR/MWAIT (and
their user-space counterparts UMONITOR/UMWAIT) to wait
for I/O completions on CXL-attached memory in a powerefficient way. Because CXL.mem provides cache-coherent
access to PMR, the processor’s monitor mechanism can directly observe device writes to completion queue entries
without explicit cache management or interrupts.
Completion and submission queues for actors are implemented as single-producer, single-consumer rings in PMR,
aligned to cache-line boundaries and mapped as cacheable
(write-back) memory. For each outstanding request, the host
performs four actions:
1. writes a descriptor into the submission ring in PMR;
2. rings a doorbell register exposed through a CXL.io BAR;
3. sets up monitoring on the cache line containing the corresponding completion entry; and
4. enters a low-power wait state using MWAIT or UMWAIT
with a bounded timeout.
When the device controller finishes processing the actor
pipeline, it performs a coherent write to the completion entry
in PMR. This write propagates through the CXL.cache/CXL.mem
protocol and updates the monitored cache line; the core’s
monitor logic detects the change and exits the wait state,
allowing the thread to resume with minimal wakeup latency.
For long-running requests, the runtime uses a hybrid strategy that combines UMWAIT with a timeout and occasional
polling to cope with the architectural limit on maximum
wait time and to avoid pathological sleep durations.
At low queue depths, MWAIT reduces CPU utilization
from 100% to 35% while maintaining comparable P99 latency.
At high queue depths, frequent wake-up overheads from
repeated MWAIT cycles erode the benefits, and traditional
polling provides higher throughput. A hybrid approach that
transitions from polling to MWAIT upon detecting empty
queues provides the best power-performance balance.

migration decisions can track transient host-side conditions

a 53.9◦ C peak while sustaining multi-GiB/s bandwidth, albeit with visible bandwidth oscillations (CV 35.99%) as the
controller trades throughput against temperature. Taken together, these results show that WIO’s benefits do not come
from a single optimization, but from the combination of byteaddressable access, coherent queue placement, PMR-backed

Elastic Thermal Management (EQ4)

As shown in Figure 1 (§2.1), SmartSSD loses 50% throughput
at 70◦ C and ScaleFlux degrades 60% at 65◦ C under sustained
writes. WIO maintains near-constant throughput by uploading actors to the host when device temperature approaches
75◦ C. The coherent PMR enables zero-copy state sharing:
actors checkpoint control state to PMR and resume on the
host without transferring working data.
This transforms the thermal constraint from a hard cliff
into an elastic trade-off: throughput varies by only a few
percent (scheduling overhead) rather than collapsing by 50–
60%. In concrete terms, WIO sustains up to 2× the throughput
of a thermally throttled SmartSSD under the same sustained
write load. When both host and device approach their limits,
the scheduler reduces the request rate rather than thrashing
between locations. Together with the minimum-residency
rule in §3, this provides a simple hysteresis guarantee: when
both sides are near saturation, WIO degrades throughput
gracefully instead of oscillating between host and device.

Computational Storage. The idea of placing computation near storage dates back to active disks in the late 1990s [1,
44], which demonstrated order-of-magnitude scan speedups
by executing user-defined functions on disk-resident data.
Modern incarnations span a wide design space, from FPGAbased platforms such as BlueDBM [28] and Samsung SmartSSD [5]
to ASIC-based drives like ScaleFlux [48]. Newport [15] and
Catalina [33] reported 2–5× speedups by moving parts of the
computation to the device and reducing data movement over
the host–device boundary, while INSIDER [45] proposed a
drive-level programming framework that exposes streaming
interfaces to host applications. However, most of these systems required application-specific modifications and relied
on non-coherent, DMA-based synchronization, which made
it difficult to generalize them across platforms and workloads.
Industry efforts to standardize CSD architectures and programming models [51] have so far not eliminated fragmentation, and commercial products still expose incompatible
interfaces [5, 48]. WIO uses CXL’s standardized coherence to

Conference’17, July 2017, Washington, DC, USA

eliminate API fragmentation, and adds the ability to reverse
compute placement dynamically—not just offload.
CXL Memory and Storage. Pooling and tiering systems
demonstrate that disaggregated memory over CXL can improve utilization and reduce DRAM provisioning in cloud
platforms. POND [32] and PolarCXLMem [59] show that
CXL-based memory disaggregation delivers substantial throughput gains and faster recovery. TPP [4] introduces transparent page placement for CXL-attached tiers, demonstrating
that the OS can manage heterogeneous memory without
application awareness. Sun et al. [53] provide the first comprehensive characterization of genuine CXL hardware, revealing latency and bandwidth behaviors that differ significantly from simulation-based projections. CXL storage research [20, 29, 34] examines the convergence of memory
and storage through byte-addressable interfaces, and shows
that eliminating software overhead helps small transfers approach device limits. WIO extends this line of work by combining a programming model, a multi-kernel architecture,
and a live migration framework on top of CXL SSDs, and
by empirically exploring the interplay between hardware
behavior, thermal constraints, and storage abstractions.
Memory Disaggregation. Disaggregated memory architectures decouple compute from memory to improve resource utilization and elasticity [49, 63]. Clio [17] co-designs
hardware and software for a disaggregated memory system with microsecond-level access, using a specialized ASIC
to handle remote memory requests without host CPU involvement. Memstrata [61] addresses CXL memory tiering
in virtualized environments, achieving 32–67% performance
improvement through careful tier management. These systems focus on memory capacity expansion; WIO complements them by adding a computation dimension to the disaggregated memory substrate, enabling storage-path logic
to execute where data resides rather than requiring data to
travel to a fixed compute location.
Near-Data Processing and SmartNICs. DPU and SmartNIC architectures support near-data computation but lack
coherent shared memory with the host [7]. iPipe [36] offloads
distributed application logic onto SmartNICs, demonstrating significant latency reduction for RPC-heavy workloads,
but its programming model is tightly coupled to the NIC’s
processing pipeline. Processing-in-memory proposals [19]
push computation into DRAM or HBM, eliminating data
movement entirely, but are limited to simple operations that
fit the memory controller’s ALU capabilities. Migrating a
running storage actor on non-coherent platforms requires
serializing the full execution state over PCIe DMA, introducing millisecond-scale stalls that make dynamic migration
impractical on the I/O fast path. WIO’s upload principle applies conceptually to any device with embedded compute,

but the zero-copy, sub-50 𝜇s migration that makes it practical depends on the coherent substrate that CXL uniquely
provides.
eBPF and Kernel-Bypass Storage. XRP [62] demonstrates
that eBPF programs attached to the NVMe driver can shortcircuit storage lookups without returning to user space,
achieving up to 2.5× throughput improvements for index
lookups. User-space storage stacks such as SPDK [25] bypass the kernel entirely, trading isolation for latency. Both
approaches assume that computation remains on the host;
WIO’s contribution is orthogonal—it moves the computation
to (or away from) the device while maintaining a standard
kernel interface. eBPF’s verifier constraints (no unbounded
loops, limited stack, no dynamic allocation) make it unsuitable for the multi-stage actor pipelines that WIO targets, but
eBPF could complement WIO by accelerating the host-side
submission and completion fast paths.
WebAssembly in Systems. WebAssembly was originally
designed as a portable compilation target for web browsers [21],
but its sandboxing properties and compact bytecode have attracted adoption in non-browser settings. Performance studies [60] show that WASM execution without SFI is within
1.08× of native for most workloads, with I/O-bound and
control-flow-heavy programs showing smaller gaps. The
Bytecode Alliance’s Wasmtime [10] and related runtimes
support ahead-of-time compilation to native code, further
reducing overhead for long-running modules. WIO is, to
our knowledge, the first system to use WebAssembly as the
execution substrate for migratable storage actors on heterogeneous host–device architectures.
Heterogeneous OS Architectures. Multi-kernel operating systems such as Barrelfish [9], Popcorn [50], K2 [35], and
Helios [40] demonstrated that running separate OS instances
on heterogeneous cores can simplify hardware-specific optimizations while maintaining a unified application interface.
WIO adopts a similar multi-kernel structure—host and device
run independent Linux instances—but focuses on a narrower
problem: migrating storage-path actors rather than general
application threads. This narrower scope allows WIO to use
a lightweight drain-and-switch protocol rather than full process migration, keeping overhead at the microsecond scale.

Conclusion

WIO demonstrates that CXL SSDs should be treated as agilityaware accelerators rather than opaque block devices or static
offload targets. By expressing storage logic as migratable
actors on a unified WebAssembly runtime over coherent
CXL.mem regions, unmodified applications benefit from
near-memory processing and byte-addressable storage. The
live migration framework and thermal-aware scheduler turn
power and thermal constraints into manageable trade-offs.
Our evaluation on production platforms confirms substantial

WIO : Upload-Enabled Computational Storage on CXL SSDs

benefits in throughput, latency, and CPU efficiency, while
thermal-aware migration eliminates the performance cliffs
that plague conventional CSDs.
The upload principle—reversible compute placement—is a
general design idea applicable beyond CXL. On coherent substrates, migration is zero-copy; on non-coherent substrates
(e.g., PCIe DMA), it would require state serialization at higher
overhead but remains conceptually valid. CXL SSDs represent a practical substrate where coherent shared memory
makes upload feasible at low cost, enabling transparent, finegrained, schedulable compute migration in the storage fast
path. As CXL hardware proliferates, agility-aware storage
designs will be essential to exploit coherent device memory and embedded compute while preserving application
simplicity.
